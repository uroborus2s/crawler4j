## 1. 问题背景

- 当前 ATM 在 `acquisition.mode=select` 下，模块环境选择回调返回 `None` 会直接把任务判成失败。
- 这适合一次性任务，不适合“固定环境池 + 长期保活服务”的业务场景。
- 典型场景是：目标并发 10，但当前只有 2 个可用环境。业务上应表现为“运行中 2、等待中 8”，而不是“2 个运行 + 8 个反复失败并重建”。
- `ctrip_crawler` 这类模块还存在“只允许复用本模块自己那批环境”的约束，宿主不能再按全局浏览器池盲抢。

## 2. 设计目标

- 把“没有可用环境”从失败改成正式的等待语义。
- 服务型作业以“服务席位”而不是“失败重试实例”来维持目标并发。
- 宿主只能在“当前模块、当前资源池、当前可接单”的环境集合里叫号。
- 模块开发者不负责排队和轮询，只负责把业务事实同步成宿主可读的资格标签。

## 3. 备选方案

| 方案 | 做法 | 优点 | 缺点 |
|---|---|---|---|
| A. 抢不到就失败，外层定时重试 | 继续沿用 `select_env -> None -> fail`，由服务层重复补单 | 改动最小 | 假失败太多，日志抖动大，不适合常驻服务 |
| B. 模块自己等待并轮询 | 模块内部 `sleep` 后重复检查资源 | 不改宿主排队模型 | 模块实例空转，占住执行生命周期，监控语义差 |
| C. 宿主统一等待队列 + 模块资源池资格标签 | 宿主负责候场、叫号、补位；模块只维护资源池资格 | 符合业务语义，支持稳定保活和容量变化补位 | 需要补充宿主契约与 SDK helper |

## 4. 选型结论

推荐方案 C。

原因：

- 它把“资源不足”建模成正常运行态，而不是异常态。
- 它保留宿主对并发、补位、终态和环境生命周期的统一治理。
- 它允许模块继续保有自己的业务判断，只是把结果翻译成宿主可读的环境资格卡片。

## 5. 已确认的关键口径

### 5.1 并发语义

- `目标并发 10` 表示需要维护 10 个服务席位，不表示必须同时存在 10 个正在执行的模块实例。
- 当只有 2 个可用环境时，正确状态是“运行中 2、等待中 8”，而不是“运行中 2、失败 8”。

### 5.2 叫号规则

- V1 固定按 FIFO 叫号。
- 环境释放后，不允许所有等待者同时抢占；宿主只从队列头部按空位数连续补位。
- 容量从 2 增长到 5 时，宿主视为“新增 3 个窗口”，会一次性把队列前 3 个席位补进去。

### 5.3 等待语义

- 等待中的席位不是失败，也不是每隔几秒重建一次模块实例。
- 主流程是“容量变化触发补位”，定时巡检只是兜底。

### 5.4 资源池隔离

- 宿主不再直接面对全局浏览器池，而是面对“当前模块的当前资源池”。
- 宿主只会从“当前模块 + 当前资源池 + 资格有效”的环境集合里分配环境。

### 5.5 资格卡片落点

- 资格卡片落在宿主内部 `env_metadata`，不是业务数据表，也不是 `core:data_table`。
- 模块开发者通过 SDK helper 写卡片，不直接操作底层持久化细节。

### 5.6 占用与黑号规则

- 环境占用后不移出资源池；资源池归属与环境是否正被占用是两件事。
- 黑号或永久失效时，先把环境标成“不可接单”，再按业务策略销毁环境或保留待人工处理。

## 6. 模块开发者需要适配什么

- 给业务上有意义的工位定义稳定的资源池名，例如 `bound_account_ready`。
- 在账号绑定完成、解绑、封禁、人工停用等业务事件点，通过 SDK helper 更新环境资格。
- 宿主重启、模块升级或批量迁移后，支持一次全量重建资源池资格快照。
- 不再自己写“没环境就等待/轮询”的逻辑。

## 7. 决策输出

- 后续设计统一采用“模块资源池资格标签 + 宿主等待队列 + FIFO 补位 + 容量变化触发补位”的组合方案。
- 资格卡片使用宿主内部 `env_metadata` 持久化；环境删除时卡片随环境级联清理。
- 下一步只进入正式设计文档，不立即实施代码。
