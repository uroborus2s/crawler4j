# 0.4.0 装饰器对象装配架构方案

**状态：** Draft  
**版本目标：** crawler4j 0.4.0  
**最后更新：** 2026-04-30  
**上游输入：** `0.4.0-decorator-object-assembly-requirements.md`、`core-native-v1` runtime descriptor、SDK CLI、`crawler4j-contracts`  
**下游输出：** Core / SDK / Contracts 重构任务、测试计划、开发者指南更新

## 1. 架构决策

0.4.0 引入 `core-native-v2` 模块运行契约。核心决策如下：

1. 代码装饰器是运行能力唯一事实源。
2. `module.yaml` 不再声明 interfaces、objects、workflows、tasks、data resources、workflow parameters。
3. Core 加载模块时扫描装饰器，生成 `ModuleRuntimeDescriptorV2`。
4. 运行模板 UI 根据 workflow 根注入对象图递归生成配置表单。
5. 任务启动时，宿主按运行模板配置创建任务环境级对象容器。
6. workflow 是被宿主实例化的对象，业务对象通过构造函数注入。
7. task 收敛为页面操作纯函数，装饰器名为 `@page_action`。
8. 数据表和命名查询也通过装饰器声明。
9. SDK 在模块打开、DevLink 注册、`check full`、manifest lock 和打包阶段前置执行装饰器诊断，阻断模块数据字段与宿主保留字段冲突。

## 2. 概念模型

| 概念 | 说明 | 生命周期 |
|---|---|---|
| `Interface` | 可被实现的能力类型，例如 `labor`、`ctrip`、`orchestrator` | 模块加载期元数据 |
| `Component` | 可被宿主实例化的业务对象类，声明 `implements`、`parameters`、`inject` | 任务环境级实例 |
| `Workflow` | 宿主实例化的 workflow 类，只通过 `inject` 接收对象 | 任务环境级实例 |
| `PageAction` | 页面操作纯函数，替代有状态 task | 调用期函数 |
| `DataTable` | 模块数据表声明 | 模块加载期元数据 |
| `DataQuery` | 模块命名查询声明 | 模块加载期元数据 |
| `ObjectContainer` | 某次 task/env 的对象图实例容器 | 任务开始到任务结束 |
| `ManifestLock` | SDK 扫描装饰器生成的只读快照 | 构建 / 安装预检 |

## 3. Contracts 重构方案

`crawler4j-contracts` 新增装饰器与元数据结构。

### 3.1 装饰器 API

```python
from crawler4j_contracts import component, data_query, data_table, interface, page_action, workflow

@interface(name="labor", label="劳保能力")
class Labor:
    pass

@component(
    name="api_labor",
    label="API 劳保对象",
    implements="labor",
    parameters=[
        {"name": "base_url", "type": "string", "required": True},
        {"name": "timeout", "type": "integer", "default": 30},
    ],
)
class ApiLabor:
    def __init__(self, base_url: str, timeout: int = 30):
        ...

@workflow(
    name="quiz_workflow",
    label="统一做题",
    inject=[
        {"name": "orchestrator", "type": "interface", "target": "orchestrator"},
    ],
)
class QuizWorkflow:
    def __init__(self, orchestrator):
        self.orchestrator = orchestrator

    async def run(self, ctx):
        return await self.orchestrator.run(ctx)

@page_action(name="open_login_page", label="打开登录页")
async def open_login_page(ctx, url: str):
    await ctx.page.goto(url)
```

### 3.2 元数据结构

装饰器只挂载元数据，不创建实例：

```python
CRAWLER4J_META_ATTR = "__crawler4j_meta__"
```

元数据最小字段：

| kind | 必填字段 | 说明 |
|---|---|---|
| `interface` | `name` | 能力接口 |
| `component` | `name`, `implements` | 可实例化对象 |
| `workflow` | `name`, `inject` | workflow 类 |
| `page_action` | `name` | 页面操作函数 |
| `data_table` | `name` | 数据表 |
| `data_query` | `name` | 命名查询 |

`InjectSpec` 采用统一列表结构：

```python
{"name": "labor", "type": "interface", "target": "labor"}
{"name": "ctrip", "type": "object", "target": "ctrip_client"}
```

含义：

- `name`: 构造函数参数名。
- `type`: `interface` 或 `object`。
- `target`: 接口名或对象名。

### 3.3 参数类型

对象创建参数沿用当前运行模板参数类型，但归属从 workflow 移到 component：

| 类型 | 用途 |
|---|---|
| `string` | 单行字符串 |
| `text` | 多行字符串 |
| `integer` | 整数 |
| `number` | 小数 |
| `boolean` | 布尔开关 |
| `enum` | 枚举选择 |

## 4. Core 重构方案

### 4.1 Runtime Descriptor V2

新增 `ModuleRuntimeDescriptorV2`：

```python
@dataclass(frozen=True)
class ModuleRuntimeDescriptorV2:
    interfaces: dict[str, InterfaceEntry]
    components: dict[str, ComponentEntry]
    workflows: dict[str, WorkflowEntry]
    page_actions: dict[str, PageActionEntry]
    data_tables: dict[str, DataTableEntry]
    data_queries: dict[str, DataQueryEntry]
    implementations: dict[str, tuple[str, ...]]
```

扫描行为：

1. import 模块包内 `interfaces/`、`objects/`、`workflows/`、`tasks/`、`data/` 下的 Python 文件。
2. 遍历 module 属性，读取 `__crawler4j_meta__`。
3. 将类/函数引用和元数据放入 descriptor。
4. 校验 interface、component、workflow、data query 名称唯一。
5. 校验 inject 目标存在。
6. 校验依赖图无环。

扫描阶段允许缓存：

- 装饰器元数据。
- 类引用。
- 函数引用。
- 依赖图。

扫描阶段禁止：

- 创建 component 实例。
- 创建 workflow 实例。
- 创建数据库连接、浏览器对象或业务状态对象。

### 4.2 对象图装配

任务启动时，Core 创建 `ObjectContainer`：

```python
class ObjectContainer:
    descriptor: ModuleRuntimeDescriptorV2
    bindings: dict[str, str]
    object_params: dict[str, dict]
    instances: dict[str, object]
```

装配流程：

1. 读取 workflow 的 `inject`。
2. 遇到 `type=interface` 时，从该 interface 的实现列表中选择对象。
3. 选择优先级：运行模板绑定 > 唯一实现 > 默认实现 > 错误。
4. 递归装配 component 的 `inject`。
5. 使用 object 参数创建叶子和中间对象。
6. 创建 workflow 实例。
7. 调用 `workflow.run(ctx)`。

构造约定：

- component 构造函数参数包括 `inject.name` 对应对象和 `parameters[].name` 对应普通值。
- workflow 构造函数只接收 `inject.name` 对应对象。
- 第一版不自动注入 `ctx` 到构造函数；需要 `ctx` 时在业务方法中传入。

### 4.3 生命周期与并发隔离

默认作用域为 `task_env`：

- 同一 task/env 内共享同一个 object container。
- 不同 task/env 不共享任何 component 或 workflow 实例。
- Core 可全局缓存 descriptor，不缓存实例。
- 任务结束后，按依赖反向顺序调用 `aclose()` / `close()`。
- 清理异常记录日志但不阻断已完成任务的终态收口。

### 4.4 运行模板 UI

运行模板从 workflow 根注入递归生成对象配置树。

UI 规则：

- interface 只有一个实现：默认选中，可折叠。
- interface 有多个实现：显示下拉选择。
- object 有 parameters：在对象节点下显示参数表单。
- object 有 inject：继续递归显示其依赖。
- 禁止用户输入任意 import path。

保存结构：

```yaml
execution:
  module: ctrip_crawler
  workflow: quiz_workflow
  object_bindings:
    orchestrator: quiz_orchestrator
    orchestrator.labor: api_labor
    orchestrator.ctrip: ctrip_client
  object_params:
    api_labor:
      base_url: https://labor.example.com
      timeout: 30
```

`RunProfile.execution.params` 在 `core-native-v2` 下不再作为 workflow 参数入口。

### 4.5 Page Action Task

`@page_action` 替代旧 task 业务脚本定位。

约束：

- 必须是函数或 async 函数。
- 第一个参数必须是 `ctx`。
- 不允许被宿主实例化为对象。
- 不承担 workflow 编排。
- 返回结果必须可归一化为 `TaskResult` 或 JSON-like dict。

workflow / orchestrator 可以通过宿主提供的能力调用 page action：

```python
await ctx.run_page_action("open_login_page", url="https://...")
```

如继续保留 `ctx.run_subtask(...)` 兼容，应在 v2 文档中标记为旧名兼容，不作为新模块推荐 API。

### 4.6 数据契约装饰器

Core 从 `@data_table` / `@data_query` 收集数据契约：

- `@data_table` 生成 data resource。
- `@data_query` 生成 named query。
- 启动时注册到现有 `ctx.db` 能力。
- `module.yaml.data` 在 v2 下不再作为正式事实源。

数据字段保留名规则：

- Core / SDK / Contracts 维护统一宿主保留字段集合，第一版至少包含 `created_at`、`updated_at`。
- SDK 同时阻断常见混淆字段 `create_at`、`update_at`，避免模块开发者把宿主维护的时间戳字段误写成业务字段。
- `@data_table` 的显式 schema、类注解推导字段、indexes 和 query output schema 不得声明或引用这些字段作为模块自有列。
- 模块确实需要保留来源系统时间戳时，应使用业务前缀字段，例如 `source_created_at`、`business_updated_at`。
- 宿主可在创建物理表时继续维护自己的系统列；模块对象图和数据契约只声明业务列。

## 5. SDK 重构方案

### 5.1 CLI 模板

新增或调整命令：

| 命令 | 作用 |
|---|---|
| `crawler4j interface create <name>` | 生成 `@interface` 模板 |
| `crawler4j component create <name> --implements <interface>` | 生成 `@component` 类模板 |
| `crawler4j workflow create <name>` | 生成 `@workflow` 类模板 |
| `crawler4j page-action create <name>` | 生成 `@page_action` 函数模板 |
| `crawler4j data table create <name>` | 生成 `@data_table` 模板 |
| `crawler4j data query create <name>` | 生成 `@data_query` 模板 |
| `crawler4j manifest lock` | 扫描装饰器并生成 `.crawler4j/manifest.lock.json` |
| `crawler4j migrate native-v2` | 生成 v1 到 v2 迁移报告或补丁 |

### 5.2 `check full`

`check full` 新增校验：

- 装饰器元数据字段合法。
- 名称为小写 snake_case。
- interface 至少有一个实现，除非显式允许抽象保留。
- workflow inject 目标存在。
- component inject 目标存在。
- object graph 无环。
- 多实现 interface 在运行模板 UI 可选择。
- object parameters 类型合法。
- page action 纯函数基本约束。
- data table 字段、索引和 query output schema 不使用宿主保留字段：`created_at`、`updated_at`、`create_at`、`update_at`。
- `module.yaml` 不含 v2 禁止字段：`workflows[].parameters[]`、`objects[]`、`interfaces[]`、`data[]` 等重复事实源。

### 5.3 打开阶段诊断

SDK 开发工具必须把模块打开阶段当成第一道质量门，而不是只在发布前校验：

| 入口 | 必须执行的诊断 |
|---|---|
| 模块项目打开 | 扫描装饰器、校验对象图、校验数据字段保留名 |
| DevLink 注册 | 与项目打开相同；阻断错误不得登记为可运行模块 |
| `crawler4j check full` | 输出完整诊断列表和文件定位 |
| `crawler4j manifest lock` | 只有诊断通过后才生成 `.crawler4j/manifest.lock.json` |
| `crawler4j package build` | 复用 lock 前后的扫描诊断，防止过期 lock 带入发布包 |

保留字段冲突属于阻断错误。错误信息应包含：

- 装饰器对象名和字段名。
- 字段来源：显式 schema、类注解、index 或 query output。
- 宿主拥有该字段的原因。
- 推荐改名方向，例如 `source_created_at`、`business_updated_at`。

### 5.4 Manifest Lock

`.crawler4j/manifest.lock.json` 是 SDK 生成文件，不是开发者事实源。

用途：

- 安装预检。
- 快速展示模块能力。
- 打包时锁定扫描结果。
- CI 中检查装饰器扫描结果是否可重复。

规则：

- 开发态以代码装饰器为准。
- 打包态必须生成 lock。
- 安装态可先读 lock，再按需 import 代码做最终校验。
- lock 与代码扫描不一致时，安装或 `check full` 失败。

## 6. Contracts / Core / SDK 分工

| 包 | 职责 |
|---|---|
| `crawler4j-contracts` | 装饰器、元数据 dataclass、参数与 inject 规范、`TaskContext` 新能力类型 |
| `crawler4j` Core | 装饰器扫描、descriptor 缓存、对象图装配、运行模板 UI、任务环境对象生命周期 |
| `crawler4j-sdk` | 模板生成、迁移命令、静态校验、manifest lock 生成、打包预检 |

## 7. 兼容与迁移

0.4.0 建议采用 `runtime_api: core-native-v2`：

- v1 模块继续由现有 descriptor 读取，作为迁移期兼容。
- v2 模块必须使用装饰器。
- v2 模块不再允许 `module.yaml.workflows[].parameters[]`。
- SDK 提供迁移报告，优先迁移 workflow/task/data 声明。

## 8. 测试策略

| 层级 | 关键测试 |
|---|---|
| Contracts | 装饰器挂载元数据、参数规范、inject 规范 |
| SDK | CLI 模板、装饰器扫描、`check full`、保留字段冲突诊断、manifest lock、v1 迁移报告 |
| Core MMS | descriptor v2 扫描、依赖图校验、循环依赖报错 |
| Core ATM | 运行模板对象图 UI、保存 `object_bindings/object_params` |
| ExecutionRunner | 每 task/env 独立对象图、workflow 实例注入、生命周期清理 |
| Data | `@data_table` / `@data_query` 注册到 `ctx.db` |
| Regression | v1 模块兼容路径、旧 workflow parameters 禁止进入 v2 |

## 9. 风险

| 风险 | 缓解 |
|---|---|
| 装饰器扫描 import 执行模块顶层代码 | SDK 模板和文档要求顶层无副作用；`check full` 对明显副作用给出警告；长期可补 AST 静态扫描 |
| 迁移范围大 | 保留 v1 runtime descriptor，v2 新模块走新契约 |
| UI 递归对象图过深 | 第一版限制最大深度并检测循环依赖 |
| 对象参数与注入参数命名冲突 | SDK `check full` 强制同一构造上下文内名称唯一 |
| SDK 与 Core 的宿主保留字段集合漂移 | 保留字段集合下沉到 Contracts 公共常量，SDK、Core 和测试共用 |
| 数据表装饰器表达力不足 | 第一版支持显式 schema，类型注解推导作为辅助 |
