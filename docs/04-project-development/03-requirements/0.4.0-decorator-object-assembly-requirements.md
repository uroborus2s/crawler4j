# 0.4.0 装饰器对象装配需求方案

**状态：** Draft  
**版本目标：** crawler4j 0.4.0  
**最后更新：** 2026-04-30  
**上游输入：** 4.0 模块开发体验讨论、现有 `core-native-v1` 运行时、SDK CLI、`crawler4j-contracts`  
**下游输出：** `docs/04-project-development/04-design/0.4.0-decorator-object-assembly-architecture.md`

## 1. 背景

当前 0.4.0 分支曾先落地 `module.yaml.workflows[].parameters[]`，用于在运行模板中为 workflow 配置普通参数。后续需求进一步明确：workflow 不应接收普通参数，也不应由模块开发者在 workflow 内根据字符串手写 registry、factory 或反射逻辑。

0.4.0 的正式方向调整为：

- 模块开发者通过装饰器声明接口、对象、workflow、页面操作和数据契约。
- 宿主扫描装饰器元数据，建立运行时对象图。
- 运行模板只配置对象实现选择和对象创建参数。
- 任务启动时，宿主为每个任务环境创建独立对象图，并把装配好的对象注入 workflow 实例。
- task 退化为页面操作纯函数，不再承担业务编排。
- `module.yaml` 不再承载运行能力对象图，只保留模块元信息和宿主级静态配置。

## 2. 目标

| ID | 目标 |
|---|---|
| `G-0400-001` | 模块开发者不再手写 `WORKFLOW = WorkflowSpec(...)` / `TASK = TaskSpec(...)` 顶层 spec 导出，改用装饰器 |
| `G-0400-002` | 模块开发者不再维护 `module.yaml` 中的 workflow 参数、对象图、数据表或 task 清单 |
| `G-0400-003` | workflow 只通过构造函数接收宿主注入对象，不接收运行参数 |
| `G-0400-004` | 对象参数只属于具体对象创建，不属于 workflow |
| `G-0400-005` | 每个任务环境有独立对象实例，避免并发任务共享状态 |
| `G-0400-006` | 页面操作 task 必须是纯函数模式，业务编排必须在 workflow/orchestrator 对象中完成 |
| `G-0400-007` | SDK 在模块打开、DevLink 注册、`check full` 和打包阶段前置阻断宿主保留数据库字段冲突 |

## 3. 范围

### 3.1 包含

- `crawler4j-contracts` 新增装饰器与元数据结构。
- Core MMS runtime descriptor 改为装饰器扫描。
- Core ATM 运行模板 UI 改为对象图配置。
- Core 执行链路新增任务环境级对象容器。
- SDK CLI 新增装饰器模板、扫描、校验、清单锁生成。
- SDK 开发工具新增模块打开阶段诊断，提前发现数据表字段与宿主保留字段冲突。
- 数据表、数据查询、workflow、component、interface、page action 均进入装饰器模式。

### 3.2 不包含

- 不支持模块开发者在运行时代码中自行反射字符串类路径。
- 不支持 workflow 级 `parameters[]` 作为正式新契约。
- 不支持默认对象实例 singleton 共享；第一版所有业务对象实例均为任务环境级。
- 不要求模块开发者手写完整 `module.yaml.objects[]`、`module.yaml.interfaces[]` 或 `module.yaml.data[]`。

## 4. 用户故事

### REQ-0400-001 装饰器声明运行能力

作为模块开发者，我希望通过 `@component`、`@workflow`、`@page_action` 等装饰器声明模块能力，以便不再理解或维护顶层 spec 导出和大型 YAML 清单。

验收标准：

- AC-1: 给定一个类使用 `@workflow(name="quiz_workflow")` 装饰，当执行 SDK `check full`，则 SDK 能识别该 workflow。
- AC-2: 给定一个类使用 `@component(implements="labor")` 装饰，当宿主加载模块，则 Core 能将它登记为 `labor` 接口的候选实现。
- AC-3: 给定同一文件同时存在多个装饰器声明，当 SDK 扫描，则每个声明都能独立进入运行时描述符。

### REQ-0400-002 module.yaml 收缩为模块级元信息

作为模块维护者，我希望 `module.yaml` 只保留模块级元信息和宿主级静态配置，以便运行能力事实源不会在 YAML 与代码之间漂移。

验收标准：

- AC-1: 给定模块代码中声明了 interfaces、components、workflows、page actions、data tables，当 `module.yaml` 不包含这些对象图时，SDK `check full` 仍能通过。
- AC-2: 给定 `module.yaml` 继续声明 `workflows[].parameters[]`，当 SDK `check full` 运行，则提示该字段在 `core-native-v2` 下不再允许。
- AC-3: 给定打包模块，SDK 能生成 `.crawler4j/manifest.lock.json` 作为扫描快照，但该文件不要求模块开发者手写。

### REQ-0400-003 宿主递归装配对象图

作为模块开发者，我希望只声明对象依赖关系，由宿主根据运行模板选择实现并递归创建对象，以便 workflow 中直接使用装配好的对象。

验收标准：

- AC-1: 给定 workflow 注入 `orchestrator` 接口，且 `quiz_orchestrator` 实现该接口，当任务启动，则宿主创建 `QuizWorkflow(orchestrator=quiz_orchestrator)`。
- AC-2: 给定 `quiz_orchestrator` 注入 `labor` 接口，且该接口有 `api_labor` 与 `web_labor` 两个实现，当编辑运行模板，则 UI 展示 `labor` 实现选择。
- AC-3: 给定 `api_labor` 声明创建参数，当用户选择 `api_labor`，则 UI 显示该对象参数表单并把结果保存到运行模板。

### REQ-0400-004 workflow 不接收参数

作为架构维护者，我希望 workflow 只依赖构造注入对象，以便 workflow 参数不会与对象创建参数混用。

验收标准：

- AC-1: 给定 workflow 装饰器声明 `parameters`，当 SDK 校验，则返回错误。
- AC-2: 给定运行模板保存对象参数，当任务启动，则这些参数只用于创建对应对象实例，不写入 workflow 参数字段。
- AC-3: 给定模块 workflow 运行时需要业务能力，则通过 `self.orchestrator` 或其他注入字段使用，不从 `ctx.runtime.params` 读取装配选择。

### REQ-0400-005 每任务环境独立对象实例

作为服务运营者，我希望并发任务之间互不共享业务对象实例，以便账号、页面、缓存和执行状态不会串任务。

验收标准：

- AC-1: 给定同一 Job 并发启动两个 task，当宿主装配对象图，则两个 task 的 `labor`、`orchestrator`、`workflow` 实例对象 ID 不同。
- AC-2: 给定同一 task 中 workflow 调用多个 page action，则这些调用共享同一个 task context，但不创建全局业务对象。
- AC-3: 给定对象实现 `close()` 或 `aclose()`，当任务结束，则宿主按依赖反向顺序清理对象。

### REQ-0400-006 task 退化为页面操作纯函数

作为模块开发者，我希望 task 只表达页面操作函数，以便业务编排集中在 workflow/orchestrator 对象中。

验收标准：

- AC-1: 给定 `@page_action` 装饰的函数，当 SDK 检查，则必须是函数或 async 函数，而不是有状态类。
- AC-2: 给定 page action 使用全局可变状态，当 SDK AST 检查能识别明显违规模式，则返回警告或错误。
- AC-3: 给定 page action 返回不可序列化对象，当运行时归一化结果，则返回明确错误。

### REQ-0400-007 数据契约装饰器化

作为模块开发者，我希望通过 `@data_table`、`@data_query` 声明数据表和查询，以便不再在 `module.yaml.data` 中重复维护数据契约。

验收标准：

- AC-1: 给定类使用 `@data_table` 装饰，当 SDK 扫描，则生成对应数据资源元数据。
- AC-2: 给定函数使用 `@data_query` 装饰，当 Core 初始化模块数据能力，则该命名查询可通过 `ctx.db.named(...)` 使用。
- AC-3: 给定 `module.yaml.data` 在 `core-native-v2` 下继续声明数据资源，当 SDK 校验，则提示迁移到装饰器或生成锁文件。

### REQ-0400-008 SDK 数据字段保留名前置校验

作为模块开发者，我希望 SDK 在模块打开、DevLink 注册、`check full`、manifest lock 和打包前识别模块数据表字段与宿主保留字段的冲突，以便 `created_at`、`updated_at`、`create_at`、`update_at` 这类字段不会等到运行时才失败。若数据表是 `managed_dataset`，SDK 还必须阻断 `record_index`、`record_key`、`run_status`、`record_status` 这些托管快照系统字段被声明为业务字段。

验收标准：

- AC-1: 给定 `@data_table` 显式 schema 声明 `created_at` 或 `updated_at`，当 SDK 打开模块、DevLink 注册或 `check full` 扫描，则返回阻断错误，并说明这些字段由宿主维护。
- AC-2: 给定 `@data_table` 类注解声明 `create_at` 或 `update_at`，当 SDK 扫描，则返回阻断错误，并建议改为业务字段名，例如 `source_created_at` 或 `business_updated_at`。
- AC-3: 给定 schema index 或 query output schema 引用被阻断字段，当 SDK 生成 manifest lock 或 package build，则失败。
- AC-4: 给定字段名不在宿主保留集合，SDK 校验通过。

### REQ-0400-009 SDK 自动脚手架与迁移

作为模块开发者，我希望 SDK 自动生成装饰器模板并提供迁移命令，以便从当前 `core-native-v1` 平滑切到 `core-native-v2`。

验收标准：

- AC-1: `crawler4j component create api_labor --implements labor` 生成带 `@component` 的类模板。
- AC-2: `crawler4j workflow create quiz_workflow` 生成带 `@workflow` 的 workflow 类模板。
- AC-3: `crawler4j migrate native-v2` 能从旧 `WORKFLOW/TASK/PageSpec` 结构生成建议迁移补丁或迁移报告。

## 5. 非功能需求

| ID | 类型 | 要求 | 验证方式 |
|---|---|---|---|
| `NFR-0400-001` | 可维护性 | 运行能力事实源只能是代码装饰器；`module.yaml` 不得重复对象图 | SDK `check full` |
| `NFR-0400-002` | 并发隔离 | 默认对象 scope 为 `task_env`，不同 task/env 不共享实例 | Core 单元测试 |
| `NFR-0400-003` | 启动性能 | Core 可缓存元数据、类引用和依赖图，但不得预创建业务对象实例 | descriptor / assembly 单测 |
| `NFR-0400-004` | 安全 | 运行模板只能选择已扫描的组件实现，不允许用户输入任意 import path | UI 与 SDK 校验 |
| `NFR-0400-005` | 兼容策略 | 0.4.0 分支允许保留 v1 读取用于迁移，但 v2 新模块默认使用装饰器 | 迁移测试 |
| `NFR-0400-006` | 开发期反馈 | SDK module-open、DevLink、`check full`、manifest lock 与 package build 必须复用同一套装饰器扫描诊断；保留字段冲突不得延后到运行时 | SDK 单测 |

## 6. 运行模板体验

运行模板表单在选择模块和 workflow 后，根据 workflow 根注入对象递归展示依赖图：

```text
一、模板信息
模块：ctrip_crawler
工作流：统一做题

对象装配
业务编排器：做题编排器
  劳保能力：[API 劳保对象 v]
    API 地址：[https://...]
    超时：[30]
  携程能力：携程对象

主体执行超时：[600] 秒
```

保存到运行模板的结构：

```yaml
execution:
  module: ctrip_crawler
  workflow: quiz_workflow
  object_bindings:
    orchestrator: quiz_orchestrator
    orchestrator.labor: api_labor
    orchestrator.ctrip: ctrip_client
  object_params:
    api_labor:
      base_url: https://labor.example.com
      timeout: 30
```

## 7. 开放问题

| ID | 问题 | 当前建议 |
|---|---|---|
| `OQ-0400-001` | 是否保留 v1 `WORKFLOW = WorkflowSpec(...)` 兼容读取 | 保留一版迁移兼容，但新模板默认只生成装饰器 |
| `OQ-0400-002` | 数据表 schema 是否优先支持类注解推导 | 第一版同时支持显式 schema 和类型注解，显式 schema 优先 |
| `OQ-0400-003` | page action 纯函数违规能否完全静态判断 | 第一版只做 AST 明显违规检查，运行时继续兜底 |
