# 6.5 CLI（脚手架与校验）

SDK CLI 的目标是让任务开发者可以“最小成本开始写脚本并在本地跑起来”，同时为后续的校验/打包/诊断预留扩展空间。

> 当前实现参考：`crawler4j_sdk/cli/commands.py`
>
> - 安装 SDK 后命令名为：`crawler4j`
> - 注意：仓库内还存在 Core 应用侧 CLI（`src/cli/*`），二者定位不同。

## 6.5.1 需求说明

- MUST 支持初始化一个独立脚本项目（可单独安装 crawler4j-sdk 并运行）。
- MUST 支持生成脚本模板（减少样板代码）。
- SHOULD 支持列出当前项目的脚本（便于选择/调试）。
- SHOULD 定义统一退出码与输出规范（利于 CI/自动化）。
- MAY 提供 validate/build/pack/doctor 等命令（阶段性落地）。

## 6.5.2 命令表（生成/校验/打包/诊断）

### 6.5.2.1 v1.0 已实现命令（以当前实现为准）

| 命令                    | 作用                           | 关键参数                                    | 产物                                                                      |
| ----------------------- | ------------------------------ | ------------------------------------------- | ------------------------------------------------------------------------- |
| `crawler4j init <name>` | 初始化脚本项目                 | `-o/--output`, `-f/--force`, `--no-install` | `pyproject.toml`、`README.md`、`debug_runner.py`、`tasks/example_task.py` |
| `crawler4j add [name]`  | 交互式创建脚本                 | `-f/--force`                                | `tasks/<name>.py`                                                         |
| `crawler4j new <name>`  | 非交互式创建脚本（兼容旧命令） | `-f/--force`                                | `tasks/<name>.py`                                                         |
| `crawler4j list`        | 列出脚本                       | 无                                          | stdout 列表                                                               |

### 6.5.2.2 预留命令（建议）

> 这部分是“架构预留”，不要求当前立刻实现，但建议预先冻结行为语义，避免未来破坏性变更。

| 命令                 | 作用                 | 说明                                                            |
| -------------------- | -------------------- | --------------------------------------------------------------- |
| `crawler4j validate` | 校验脚本/项目结构    | 语法检查、name/display_name 必填、依赖检查、配置 schema（如有） |
| `crawler4j build`    | 构建可发布产物       | 例如生成 wheel/sdist（若脚本项目需要发布）                      |
| `crawler4j pack`     | 打包为模块可安装格式 | 对齐 Core 的模块安装器（见 07-Modules / Core）                  |
| `crawler4j doctor`   | 诊断环境             | Playwright 浏览器安装、权限、网络连通性等                       |

## 6.5.3 输出与退出码规范

### 6.5.3.1 输出规范

- SHOULD：
  - 普通信息输出到 stdout
  - 错误信息输出到 stdout 或 stderr 均可，但必须有清晰前缀（如 `❌`）
  - 对关键产物路径给出明确提示（便于 IDE/脚本继续处理）

### 6.5.3.2 退出码

- MUST：
  - `0`：成功
  - `1`：可预期失败（参数错误、目标目录已存在且未 `--force`、文件写入失败等）
- SHOULD：未来可扩展细分（如 `2`=校验失败，`3`=依赖缺失），但同一 major 内不得改变既有含义。
