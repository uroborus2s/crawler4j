# 6.2 最小验收清单

在把模块交给别人之前，至少完成下面这些检查。

如果你是第一次交付模块，建议真的把这页当作 checklist 一项一项过，不要只靠印象判断“应该差不多了”。

## 开发阶段检查

1. `init-model` 能在干净目录里重新回放
2. 模块目录里存在 `module.yaml`、根 `__init__.py`、`tasks/`、`workflows/`
3. `uv run crawler4j list` 能列出任务
4. `module.yaml.workflows` 与 `workflows/` 中的真实工作流保持一致

### 这组检查在验证什么

它验证的是：你的模块项目本身是不是结构完整、最基本的自发现是不是成立。

## 调试阶段检查

1. 模块能被注册成 `DevLink`
2. 策略中的 `execution.module` 正确指向 `module.yaml.name`
3. 策略中的 `execution.workflow` 正确指向目标工作流
4. ATM 中能看到 `🐞 调试`
5. IDE 能附加到 `debugpy` 并命中断点

### 这组检查在验证什么

它验证的是：你的开发链路是不是通的。如果这里没有通过，就不要急着做正式交付，因为你还没证明宿主能稳定理解你的模块。

## 正式交付检查

1. zip 中只有一个根目录
2. zip 根目录里包含 `module.yaml`
3. 模块能在宿主中成功安装
4. 安装后来源切换为正式安装模块
5. 安装后仍能完成一次 smoke

### 这组检查在验证什么

它验证的是：你的交付物是不是可安装、可识别、可运行。这一步通过，才说明你的模块不只是“开发时能跑”，而是“交出去也能跑”。

## 建议追加的人工检查

如果模块要交给其他人使用，建议额外确认：

1. README 或交付说明里写清楚模块用途
2. 运行参数含义明确
3. 外部依赖是否已在宿主环境准备好
4. 常见失败场景是否有清晰日志或截图

完成这些后，才可以认为这个模块具备基本交付质量。

## 一条适合小白的使用方式

第一次交付时，建议你真的把它当成下面三关：

1. 结构关：模块目录和清单有没有问题
2. 调试关：宿主能不能正确调试它
3. 安装关：zip 能不能正确安装并执行

这三关都过了，再说“交付完成”会更稳。
