# 6.1 zip 打包与正式安装

## 为什么正式交付必须走 zip

当前模块管理的正式安装链路认的是 zip 包。  
安装动作本质上是：

1. 选择 zip
2. 校验 zip 结构
3. 解压到应用受控目录
4. 重新加载模块注册表

所以当前正式交付物不是 wheel，而是 zip。

这条规则对新手很重要，因为它会直接影响你的交付动作。如果你把“我已经能 build”误认为“我已经能交付”，你很容易在最后一步才发现正式安装链路根本没走通。

## zip 的最稳结构

```text
hotel_demo-1.0.0.zip
└── hotel_demo/
    ├── __init__.py
    ├── module.yaml
    ├── config_schema.json
    ├── tasks/
    └── workflows/
```

最关键的结构要求只有两条：

1. zip 中只能有一个根目录
2. 根目录里必须能找到 `module.yaml`

### 小白最容易打错的结构

最常见的错误有两类：

1. zip 里直接铺平了一堆文件，没有根目录
2. zip 里套了多层目录，导致真正的 `module.yaml` 不在第一层根目录里

所以打包之后，建议你自己先解压看一眼结构，不要完全靠猜。

## 推荐打包命令

在模块目录的父目录执行：

```bash
uv run python -m zipfile -c hotel_demo-1.0.0.zip hotel_demo
```

不要在模块目录内部执行这个命令，否则 zip 里的根目录层级很容易不对。

### 为什么强调“在父目录执行”

因为这个命令的最后一个参数是目录名。你在父目录执行时，zip 里会自然得到一个标准根目录。你在模块目录内部执行时，压出来的结构很容易不符合宿主预期。

## 安装时会发生什么

如果宿主里已经存在同名 `DevLink`，当前正式安装会自动移除那条开发链接。  
这么做的原因是避免运行时继续优先落回本地源码目录。

安装完成后，你通常会看到：

1. 模块来源从“开发链接”切换成正式安装模块
2. 同名作业的 `🐞 调试` 入口消失

这不是 bug，而是当前设计。

### 这对新手意味着什么

你不要把“安装后调试按钮消失”误判成安装失败。它更可能只是说明：

- 你的模块已经不再来自 `DevLink`
- 当前宿主正在把它当作正式安装模块运行

## 正式安装后应该验证什么

最少验证下面四件事：

1. 模块能在模块列表里被识别
2. 工作流仍然可被策略选中
3. 模块运行时来源确实变成安装目录
4. 任务能完成一次 smoke

如果你只验证了“zip 能装进去”，但没有验证“安装后还能跑”，这次交付还不能算完成。

## 一个适合小白的正式交付心法

正式交付时，不要只问：

> “这个包能不能装进去？”

还要问：

1. 装进去后，宿主是不是认对了来源？
2. 策略还能不能找到它？
3. 工作流还能不能执行？
4. 运行结果是否和 DevLink 时一致？

这 4 个问题都过了，才更接近真正的“交付完成”。
