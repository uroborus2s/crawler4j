# 1.1 系统地图与术语

## 先建立一张正确的地图

开发 `crawler4j` 模块时，你面对的不是单一 Python 包，而是一条“CLI 脚手架 -> 模块项目 -> 宿主 Core -> 策略与作业 -> 调试/安装验收”的完整链路。

先记住下面这些角色：

| 对象 | 你在哪会遇到它 | 它负责什么 |
|---|---|---|
| `crawler4j-sdk` CLI | 终端 | 创建模块骨架、补任务、补工作流、补 UI |
| 模块项目 | 你的源码仓库 | 放 `module.yaml`、`__init__.py`、`tasks/`、`workflows/` |
| `module.yaml` | 模块根目录 | 模块清单，声明名称、版本、工作流、UI 扩展 |
| 模块根 `__init__.py` | 模块根目录 | Core 最终导入的运行入口 |
| Core / 桌面应用 | `crawler4j` 主程序 | 扫描、加载、调试、安装和执行模块 |
| Strategy | 宿主策略配置 | 决定运行哪个模块、哪个工作流、带什么参数 |
| Job / ATM | 自动化任务管理 | 让真实作业运行或进入调试 |
| DevLink | 模块管理中的开发链接 | 把本地源码目录接进 Core |
| zip 安装包 | 正式交付物 | 给宿主应用做正式安装与验收 |

## 用一句大白话解释这套系统

如果你完全是第一次接触，可以先把这套系统想成下面这样：

- `crawler4j-sdk` CLI：帮你生成和补齐模块项目骨架
- 模块项目：你真正写代码的地方
- 宿主 `crawler4j`：真正执行你模块的地方
- Strategy / Job / ATM：告诉宿主“运行哪个模块、哪个工作流、带什么参数”
- DevLink：开发模式下，把宿主临时指向你的本地源码目录
- zip：正式交付时，把你的模块以安装包形式交给宿主

也就是说：

> 你平时写的是模块项目，但真正跑起来的是宿主应用里的执行链。

## 当前推荐心智模型

把模块理解成“由宿主 Core 负责调度和运行的 Python 包”，不要把它理解成“一个可以自己独立启动的应用”。

对模块开发者来说，最关键的三个事实是：

1. 你真正交付给 Core 的，是一个包含 `module.yaml` 和根 `__init__.py` 的目录。
2. Core 不是从 wheel 元数据里发现模块，而是从模块目录直接加载它。
3. 模块最终运行在宿主 `crawler4j` 的 Python 环境里，不是运行在你模块项目自己的虚拟环境里。

如果你把第 3 点忘了，后面几乎一定会碰到“本地能 import，宿主里却报错”的问题。

## 从源码到运行的完整链路

```text
你写模块源码
-> 用 CLI 生成/补齐脚手架
-> 得到一个包含 module.yaml 的模块目录
-> 通过 DevLink 或 zip 让 Core 看到这个目录
-> 策略把 execution.module / execution.workflow 指向这个模块
-> ATM / DebugService 发起真实执行
-> ExecutionRunner 调用模块 hooks、TaskFlow、TaskScript
```

这条链路里，真正影响运行结果的关键节点只有几个：

- `module.yaml.name`
- 根 `__init__.py` 暴露的 `run(context)` 和可选 hooks
- `workflows[*].name`
- 策略里的 `execution.module` 与 `execution.workflow`

换句话说，新手最应该优先保证的不是“代码很多”，而是这四个点都写对。

## 新手视角下，一次完整开发会发生什么

如果把整个过程拆成最常见的几步，大概是这样：

1. 你先用 CLI 生成一个模块目录
2. 你在 `tasks/` 里写任务脚本
3. 你在 `workflows/` 里写工作流
4. 你把工作流信息写进 `module.yaml`
5. 你把模块注册成 `DevLink`
6. 你在宿主策略里填写 `execution.module` 和 `execution.workflow`
7. 你在 ATM 里调试或运行
8. 你打 zip 做正式安装验收

每一步都不是孤立的。前一步写错，后一步通常就会表现为“看起来像没反应”。

## 作为模块作者，你最终要交付什么

一个最小可交付模块至少包含：

```text
your_module/
├── __init__.py
├── module.yaml
├── tasks/
│   ├── __init__.py
│   └── some_task.py
└── workflows/
    ├── __init__.py
    └── some_workflow.py
```

如果你需要模块配置页，通常还会包含：

```text
config_schema.json
```

如果你需要代码型页面扩展，通常还会包含：

```text
ui.py
```

但对第一次开发模块的人，不建议从自定义 UI 开始。先把任务、工作流、DevLink 调试和 zip 安装跑通，才是正确顺序。

## 这张地图里最容易误会的地方

### 误会 1：我写的是 Python 项目，所以直接运行它就够了

不够。模块项目只是源码载体。真正的目标不是“这个目录里有 Python 文件”，而是“宿主能识别并执行这个目录”。

### 误会 2：CLI 创建成功，就说明模块已经能跑

也不够。CLI 只能保证你获得了一个比较标准的起点，不保证：

- 策略已经写对
- 宿主已经识别它
- 调试链路已经通
- 正式安装已经没问题

### 误会 3：DevLink 和正式安装是同一件事

不是。DevLink 是开发模式，zip 安装是交付模式。前者解决“我在本地改代码，宿主怎么调到它”，后者解决“别人怎么安装和使用它”。

## 开发时最容易混淆的三组概念

### `model` vs `module`

- `model` 是 CLI 历史命名，体现在 `crawler4j init-model`。
- `module` 是运行时正式命名，体现在 `module.yaml`、模块管理、安装、加载、调试。

实际开发时，你可以把它们理解成同一件事。

### 模块项目 vs 宿主应用

- 模块项目是你写代码的地方。
- 宿主应用是最终加载、执行、调试和安装模块的地方。

模块项目能 `uv run crawler4j ...`，不代表宿主已经认识你的模块。要让宿主真正加载它，你还需要 DevLink 或正式安装。

### 任务脚本 vs 工作流

- `TaskScript` 是原子任务，完成一个具体动作。
- `TaskFlow` 是编排器，负责决定子任务执行顺序、循环、分支和阶段状态。

初学者最稳的做法是：先写一个任务脚本，再写一个只调用这个任务脚本的最小工作流。
