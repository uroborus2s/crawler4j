# 使用说明

当前宿主应用的核心操作面板只有 6 个：`📊 仪表盘`、`📋 任务监控`、`🖥️ 环境管理`、`📦 模块管理`、`⚙️ 策略配置`、`🔧 系统设置`。第一次使用时，不需要每个页面都深入，先按一条完整链路走通更重要。

## 1. 首次使用建议顺序

1. 先到 `🔧 系统设置` 确认网络、浏览器和日志配置。
2. 再到 `📦 模块管理` 确认当前模块来源和状态。
3. 然后到 `⚙️ 策略配置` 绑定模块与工作流。
4. 最后到 `📋 任务监控` 创建作业、运行或调试。

## 2. 正式模块的典型使用路径

如果你拿到的是别人交付的模块 zip，推荐按下面顺序操作：

1. 打开 `📦 模块管理`。
2. 点击 `📥 安装模块`，选择模块 zip。
3. 确认模块已经出现在列表中，且来源是正式安装模块。
4. 打开模块详情，补齐模块配置。
5. 打开 `⚙️ 策略配置`，把 `execution.module` 指向 `module.yaml.name`，把 `execution.workflow` 指向目标工作流。
6. 打开 `📋 任务监控`，创建或选择绑定该策略的作业并启动。

正式安装模块会被解压到应用数据目录下的 `modules/` 中，而不是放在仓库根目录的 `modules/`。

## 3. DevLink 联调的典型路径

如果你是在本地开发模块并想联调：

1. 打开 `📦 模块管理`。
2. 点击 `🔗 添加开发模块`，选择包含 `module.yaml` 的模块根目录。
3. 刷新后确认来源显示为开发链接（DevLink）。
4. 在 `⚙️ 策略配置` 中正确填写 `execution.module` 和 `execution.workflow`。
5. 在 `📋 任务监控` 中创建作业并观察是否出现 `🐞 调试`。

当前调试链只对 DevLink 模块开放。
如果你已经把同名模块正式安装进去，安装过程会移除同名 DevLink，随后调试按钮消失是当前设计，不是 bug。

## 4. 各页面最值得先看的内容

### 仪表盘

- 当前运行中的任务数
- 环境数
- 模块数
- 当前工作区版本

### 环境管理

- 当前浏览器环境是否可用
- 代理与运行环境是否正常

### 模块管理

- 模块来源：内置 / 正式安装 / DevLink
- 模块状态：启用 / 禁用 / 不兼容 / 无效
- 模块配置页是否能正常打开

### 策略配置

- `execution.module`
- `execution.workflow`
- 运行参数是否和模块当前版本匹配

### 任务监控

- 作业是否绑定了正确策略
- 触发规则是否符合预期
- 当前状态是运行中、暂停、完成还是异常

## 5. 常见问题

### 看得到模块，但没有 `🐞 调试`

优先检查：

1. 模块来源是不是 DevLink。
2. 作业是否已经绑定策略。
3. 策略里的 `execution.module` 是否等于 `module.yaml.name`。
4. 策略里的 `execution.workflow` 是否等于目标工作流名。

### 安装了 zip，但行为还像本地源码

优先回到 `📦 模块管理` 检查模块来源，并手动刷新模块列表。
当前正确的正式安装状态应该表现为：模块来源是正式安装模块，同名 DevLink 已被移除。

### 模块能识别，但工作流选不到

优先核对：

1. `module.yaml` 里是否已经声明对应工作流。
2. 工作流名字是否与策略里的 `execution.workflow` 一致。
3. 模块列表是否已经刷新到最新状态。
