# 使用说明

当前宿主应用的核心操作面板只有 5 个：`📊 仪表盘`、`📋 任务监控`、`🖥️ 环境管理`、`📦 模块管理`、`🔧 系统设置`。第一次使用时，不需要每个页面都深入，先按一条完整链路走通更重要。

## 1. 首次使用建议顺序

1. 先到 `🔧 系统设置` 确认网络、浏览器和日志配置。
2. 再到 `📦 模块管理` 确认当前模块来源和状态。
3. 然后到 `📋 任务监控` 创建任务并填写运行模板。
4. 最后在同一页按任务模式执行一次、启动、暂停或调试。

## 2. 正式模块的典型使用路径

如果你拿到的是别人交付的模块 zip，推荐按下面顺序操作：

1. 打开 `📦 模块管理`。
2. 点击 `📥 安装模块`，选择模块 zip。
3. 确认模块已经出现在列表中，且来源是正式安装模块。
4. 打开模块详情，补齐模块配置。
5. 打开 `📋 任务监控`，创建任务并把运行模板中的 `execution.module` 指向 `module.yaml.name`，把 `execution.workflow` 指向目标工作流。
6. 根据业务选择“批次任务”或“持续保活”，再执行一次或启动该任务并观察运行状态。

正式安装模块会被解压到应用数据目录下的 `modules/` 中，而不是放在仓库根目录的 `modules/`。

## 3. DevLink 联调的典型路径

如果你是在本地开发模块并想联调：

1. 打开 `📦 模块管理`。
2. 点击 `🔗 添加开发模块`，选择包含 `module.yaml` 的模块根目录。
3. 刷新后确认来源显示为开发链接（DevLink）。
4. 在 `📋 任务监控` 中正确填写运行模板的 `execution.module` 和 `execution.workflow`。
5. 观察该任务是否出现 `🐞 调试`。

当前调试链只对 DevLink 模块开放。
如果你已经把同名模块正式安装进去，安装过程会移除同名 DevLink，随后调试按钮消失是当前设计，不是 bug。

## 4. 各页面最值得先看的内容

### 仪表盘

- 当前运行中的任务数
- 环境数
- 模块数
- 当前工作区版本

### 环境管理

- 当前浏览器环境是否可用
- 代理与运行环境是否正常
- 创建环境时，环境名称输入框会先展示系统建议名（如 `env-YYYYMMDD-N`），你可以直接修改；保持默认则继续按系统序号创建
- 手动创建成功后，环境会直接保持在“运行中 / 已连接”状态；只有点击停止后，才会关闭运行环境并回到“就绪”
- 在环境列表里点击指纹浏览器环境的 `▶` 启动时，宿主会先按需拉起 BitBrowser / VirtualBrowser；如果路径未配置、路径失效或启动超时，会直接提示错误
- 如果浏览器窗口已经打开，但自动化连接还没完成，状态会先显示为“启动中”，而不是错误地回退成“就绪”
- 对于新版 VirtualBrowser 只返回 `debuggingPort` 的启动响应，宿主现在会自动转换为可用的 CDP 调试入口，不需要手工改配置
- 如果宿主返回的是 HTTP 调试入口，宿主现在会先主动请求 `/json/version` 或 `/json/list`，解析出真实的 `webSocketDebuggerUrl` 后再交给 Playwright，绕过部分宿主对 `/json/version/` 的兼容问题
- 对于只接受 `localhost` 或只接受 `127.0.0.1` 的调试入口，宿主现在也会自动补试另一侧的 HTTP 调试地址，并优先使用探测到的真实 websocket 端点连接
- 删除指纹浏览器环境时，宿主会先检查对应浏览器 API 是否可用；只有外部环境删除成功后，才会移除本地数据库记录

### 模块管理

- 模块来源：内置 / 正式安装 / DevLink
- 模块状态：启用 / 禁用 / 不兼容 / 无效
- 模块配置页是否能正常打开

### 任务监控

- 作业是否已经配置了正确运行模板
- `execution.module`
- `execution.workflow`
- `批次任务` 支持两种触发方式：
  - `执行一次`：点击列表中的 `▶ 执行一次` 后立即发起一批任务；如果上一批还没结束，不会再次发起
  - `Cron 定时`：点击 `▶ 启动` 后交给调度器按 Cron 规则触发批次
- `持续保活` 固定为手动启动，启动后会一直维持目标并发，直到点击 `⏸ 暂停`
- 当前状态是运行中、暂停、完成还是异常

## 5. 常见问题

### 看得到模块，但没有 `🐞 调试`

优先检查：

1. 模块来源是不是 DevLink。
2. 作业是否已经配置运行模板。
3. 运行模板里的 `execution.module` 是否等于 `module.yaml.name`。
4. 运行模板里的 `execution.workflow` 是否等于目标工作流名。

### 安装了 zip，但行为还像本地源码

优先回到 `📦 模块管理` 检查模块来源，并手动刷新模块列表。
当前正确的正式安装状态应该表现为：模块来源是正式安装模块，同名 DevLink 已被移除。

### 模块能识别，但工作流选不到

优先核对：

1. `module.yaml` 里是否已经声明对应工作流。
2. 工作流名字是否与运行模板里的 `execution.workflow` 一致。
3. 模块列表是否已经刷新到最新状态。
