# 配置说明

当前项目里真正会影响运行结果的配置主要分成三层：系统设置、模块设置、执行配置。第一次接手时，先把这三层分清，比一次性记很多字段更重要。

## 1. 系统设置

系统级配置在应用内的 `🔧 系统设置` 页面维护，当前分为三类：

### 通用

- 主题
- 语言
- 自动检查更新
- 开机自启动
- 启动时最小化

### 网络

- 代理模式：跟随系统 / 不使用代理 / 手动配置
- HTTP 代理地址
- BitBrowser API 端口与程序路径
- VirtualBrowser API 端口、API Key 与程序路径

### 资源

- 日志级别
- 日志保留天数

这些设置会持久化到应用数据目录下的 `config.db`，不会写回仓库源码。

## 2. 哪些设置改完需要重启

当前至少有两类设置按实现需要重启后再完全生效：

- 语言
- 代理模式

如果页面提示“部分设置需要重启应用后生效”，按提示重启即可，不要把它误判成保存失败。

## 3. 模块设置

模块相关配置不是在全局设置页维护，而是在 `📦 模块管理` 的模块详情页中维护。

当前模块配置由宿主统一维护并持久化，主要分成两层：

- 模块级配置：按模块名保存。
- 工作流级配置：按模块名和工作流名保存。

这些内容同样会写入 `config.db`，而不是保存在模块项目目录里。
当前正式落点是 `config.db.module_config_entries`。
`config_schema.json` 已不再作为正式配置入口。
当前模块详情页的 `配置` 标签会直接提供：

- 模块级 YAML 配置编辑器
- 工作流选择器
- 当前工作流对应的 YAML 覆盖编辑器
- 恢复模块默认 / 恢复 Workflow 默认按钮

模块运行时通过 `ctx.get_config()` 读取到的，只是这里保存的模块配置与工作流覆盖结果；
不会再混入 `workflow`、`devel_mode`、调试参数或环境创建参数。

如果模块在 `module.yaml` 中声明了 `config_defaults`：

- 宿主第一次加载这个模块时，会把 `config_defaults.module` 和 `config_defaults.workflows` 初始化到 `module_config_entries`
- 这个初始化只做一次
- 后续模块升级、刷新、重扫不会自动重置已有配置
- 详情页里的恢复默认按钮会读取当前 `module.yaml.config_defaults`，弹出警告确认后再覆盖对应作用域

这意味着：

- `module.yaml.config_defaults` 是“初始模板”
- `config.db.module_config_entries` 才是“当前真实配置”
- 模块运行时仍然只读数据库里的有效配置，不直接读 `module.yaml`

### 关于自定义页面的当前限制

如果模块使用代码型 UI 页面（`ui:SomePage`），要注意当前并不是所有来源都允许直接加载：

- 内置模块和 DevLink 模块默认允许。
- 外部正式安装模块只有在命中 allowlist 时才会被加载。

因此第一次接手时，优先把“模块设置保存成功 + 正常运行链路”确认好，不要一开始就把问题定位到复杂 UI 上。

## 4. 执行配置

真正决定“运行哪个模块、哪个工作流”的，不是系统设置页，而是任务自身运行模板里的执行字段：

- `execution.module` 必须等于目标模块的 `module.yaml.name`
- `execution.workflow` 必须等于目标工作流名
- `execution.params` 是运行模板默认输入
- `job.params` 是当前作业的一次性覆盖输入

当前运行模板的资源配置已收敛成两条主路径：

- `创建环境`：基础信息和执行配置放在同一页完成，直接选择“模块 + 脚本”作为执行脚本，再填写环境类型、Provider。默认推荐 `virtualbrowser`。
- `选择环境`：先选择模块里声明的环境选择回调函数。该模式不再填写 Provider 和匹配规则，宿主会把当前全部 `ready` 浏览器环境作为候选集传给模块回调，由回调返回目标环境 ID。
- 执行脚本下拉当前会优先展示工作流 `display_name`；真正写入运行模板并参与调度的仍然是 `execution.workflow=name`。

关于 `选择环境` 还需要记住两点：

- 回调函数没有默认值；切到该模式后必须手动选择一个模块回调。
- 如果界面里选择的是脚手架自带的 `返回 None` 占位回调，表单会直接提示“函数返回了 none”；如果仍然保存并执行，任务会在运行时直接失败。

如果 `Provider=virtualbrowser`，创建环境页里的指纹参数已经按官方创建页交互做成显式表单，不再要求先写一大段 YAML。当前已支持的常见项包括：

- `浏览器版本` 下拉选择，当前项目默认版本为 `145`
- `内核版本` 自动匹配
- `User Agent` 默认/自定义/随机；新建“创建环境”时默认就是自定义随机 UA，随机按钮会按当前选中的浏览器大版本在本地生成 UA 值，并作为自定义参数提交
- `Sec-CH-UA` 默认/自定义品牌列表，支持按 `brand + version` 增删条目
- 语言、时区改为下拉选择，并保留“跟随 IP”开关；地理位置仍支持跟随 IP 或自定义
- `WebRTC`、地理位置权限、字体 改成截图同款的分段按钮交互；分辨率改成“跟随电脑 / 自定义”分段按钮，选择自定义后使用预设分辨率下拉
- `Canvas`、`WebGL 图像`、`AudioContext`、`ClientRects`、`Speech Voices` 改为按钮式的默认/随机切换；其中新建 `virtualbrowser` 时，`AudioContext` / `ClientRects` / `Speech Voices` 默认就是“随机”
- `WebGL` 元数据改为按钮式的默认/自定义；选择自定义后会显示 `WebGL 厂商` 和 `WebGL 渲染` 的选项式输入
- `WebGPU` 提供按钮式的默认 / 基于 WebGL 切换
- `CPU`、`内存` 使用数字输入；新建 `virtualbrowser` 时默认是 `4 核 / 8 GB`
- `设备名称`、`MAC 地址` 改为截图同款的“默认 / 自定义”按钮，并支持快速生成；新建时默认就是自定义随机值，设备名不再固定 `MacBook-` 前缀，而是 `10` 位随机前缀字符加 UUID 片段
- `Do Not Track`、`硬件加速` 改成开关样式交互
- `SSL`
- `端口扫描保护` 与白名单；`SSL` / `端口扫描保护` / `启动参数` 统一改成截图同款的按钮式开关，不再使用下拉框
- `启动参数` 默认/自定义

如果 IP 绑定方式选择“从 IP 池绑定”，当前运行模板会把绑定策略一起写入创建参数，至少支持：

- `最少绑定数`：优先分配当前绑定数量最少的 IP。
- `最高安全度`
- `最长有效期`

当前新建 `virtualbrowser` 创建配置时，`随机化指纹` 会默认开启；模板只保存随机化开关和随机模式，真正创建环境时才展开出本次使用的随机 `UA`、随机设备名、随机 `MAC`，因此同一模板连续创建两次也会得到不同值。`字体`、`Canvas`、`WebGL 图像`、`AudioContext`、`ClientRects`、`Speech Voices` 只要在模板里选了“随机”，就会直接作为本次创建的随机指纹参数下发；旧的 `__randomize_after_create__` 方案已完全移除，不再兼容。

如果这两个字段写错，常见表现是：

- 模块能看到，但作业跑不起来
- 工作流文件存在，但任务运行模板里选不到
- `🐞 调试` 按钮不出现

当前执行期参数的边界也要记住：

- `execution.params`、`job.params` 会保存在 `state.db.jobs` 中
- 调度时它们会进入 `ctx.runtime["execution_params"]`、`ctx.runtime["job_params"]`
- 合并后的有效输入会放到 `ctx.runtime["params"]`
- `workflow`、`devel_mode`、`creation_params` 也都属于 `ctx.runtime`
- `ctx.get_config()` 不再读取这些执行态字段

## 5. 配置存放位置速查

| 内容 | 位置 |
|---|---|
| 系统设置 | `config.db` 的 `settings` 表 |
| 模块/工作流设置 | `config.db` 的 `module_config_entries` 表 |
| 模块数据表 schema / records | `data.db` |
| 运行态状态、KV、环境锁 | `state.db` |
| 日志 | `<app-data>/logs/` |
| 正式安装模块 | `<app-data>/modules/` |

`<app-data>` 的平台路径见 [安装说明](./installation.md)。

如果是从旧版本升级，需要注意：当前运行时代码不会自动把历史 `state.db.kv_store` 里的模块数据表 schema / dataset records 搬到 `data.db`。如果旧环境里仍有这类数据，需要显式迁移工具或人工导入；当前页面只会读取 `data.db`。

## 6. 最常见的配置错误

- 代理模式设成“手动配置”，但没有填写有效的 HTTP 代理地址。
- 指纹浏览器端口或程序路径填写了历史值，当前机器上并不存在对应程序。
- 模块项目里新增了依赖，但宿主运行环境没有同步具备该依赖。
- 运行模板里填写了显示名、目录名或 zip 文件名，而不是 `module.yaml.name`。
