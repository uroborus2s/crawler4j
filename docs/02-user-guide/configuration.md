# 配置说明

当前项目里真正会影响运行结果的配置主要分成三层：系统设置、模块设置、执行配置。第一次接手时，先把这三层分清，比一次性记很多字段更重要。

## 1. 系统设置

系统级配置在应用内的 `🔧 系统设置` 页面维护，当前分为三类：

### 通用

- 主题
- 语言
- 自动检查更新
- 开机自启动
- 启动时最小化

### 网络

- 代理模式：跟随系统 / 不使用代理 / 手动配置
- HTTP 代理地址
- BitBrowser API 端口与程序路径
- VirtualBrowser API 端口、API Key 与程序路径

### 资源

- 日志级别
- 日志保留天数

这些设置会持久化到应用数据目录下的 `config.db`，不会写回仓库源码。

## 2. 哪些设置改完需要重启

当前至少有两类设置按实现需要重启后再完全生效：

- 语言
- 代理模式

如果页面提示“部分设置需要重启应用后生效”，按提示重启即可，不要把它误判成保存失败。

## 3. 模块设置

模块相关配置不是在全局设置页维护，而是在 `📦 模块管理` 的模块详情页中维护。

当前模块配置有两种常见形式：

- 声明式配置：模块提供 `config_schema.json`，宿主生成配置表单并保存。
- 工作流级配置：宿主按模块名和工作流名持久化配置。

这些内容同样会写入 `config.db`，而不是保存在模块项目目录里。

### 关于自定义页面的当前限制

如果模块使用代码型 UI 页面（`ui:SomePage`），要注意当前并不是所有来源都允许直接加载：

- 内置模块和 DevLink 模块默认允许。
- 外部正式安装模块只有在命中 allowlist 时才会被加载。

因此第一次接手时，优先把“声明式配置 + 正常运行链路”确认好，不要一开始就把问题定位到复杂 UI 上。

## 4. 执行配置

真正决定“运行哪个模块、哪个工作流”的，不是系统设置页，而是任务自身运行模板里的执行字段：

- `execution.module` 必须等于目标模块的 `module.yaml.name`
- `execution.workflow` 必须等于目标工作流名

如果这两个字段写错，常见表现是：

- 模块能看到，但作业跑不起来
- 工作流文件存在，但任务运行模板里选不到
- `🐞 调试` 按钮不出现

## 5. 配置存放位置速查

| 内容 | 位置 |
|---|---|
| 系统设置 | `config.db` 的 `settings` 表 |
| 模块/工作流设置 | `config.db` 的 `configs` 表 |
| 运行态状态、KV、环境数据 | `state.db` |
| 日志 | `<app-data>/logs/` |
| 正式安装模块 | `<app-data>/modules/` |

`<app-data>` 的平台路径见 [安装说明](./installation.md)。

## 6. 最常见的配置错误

- 代理模式设成“手动配置”，但没有填写有效的 HTTP 代理地址。
- 指纹浏览器端口或程序路径填写了历史值，当前机器上并不存在对应程序。
- 模块项目里新增了依赖，但宿主运行环境没有同步具备该依赖。
- 运行模板里填写了显示名、目录名或 zip 文件名，而不是 `module.yaml.name`。
