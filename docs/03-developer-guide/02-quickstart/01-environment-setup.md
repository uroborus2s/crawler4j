# 2.1 开发前准备

## 最小前置条件

开始前，你至少需要下面四样东西：

1. Python `3.12+`
2. `uv`
3. `crawler4j-sdk` CLI
4. 一个可运行的 `crawler4j` 宿主应用，用于 DevLink 调试和 zip 安装验收

其中前 3 项是模块项目开发环境，第 4 项是运行与验收环境。

如果你目前只有前 3 项，没有第 4 项，那么你可以先完成“创建模块项目”和“写代码”这部分，但暂时无法完成真实调试和正式安装验收。

## 先做环境自检

在终端里先确认这些命令可用：

```bash
python3 --version
uv --version
uvx --from crawler4j-sdk crawler4j --help
```

如果这三个命令里有任何一个失败，请先不要往后走。对新手来说，最省时间的方式是：先把环境问题解决掉，再开始看模块开发步骤。

如果你打算长期开发模块，也可以把 CLI 装成工具：

```bash
uv tool install crawler4j-sdk
crawler4j --help
```

如果你只想一次性使用，继续用 `uvx` 即可：

```bash
uvx --from crawler4j-sdk crawler4j --help
```

## 三种常见调用方式

第一次接触时，你不需要纠结“到底哪种方式最高级”。先记住：三种方式都能用，只是适合场景不同。

### 方式 1：一次性调用

适合第一次体验或临时脚手架。

```bash
uvx --from crawler4j-sdk crawler4j init-model hotel_demo
```

### 方式 2：安装成全局工具

适合长期维护多个模块项目。

```bash
uv tool install crawler4j-sdk
crawler4j init-model hotel_demo
```

### 方式 3：进入模块项目后用 `uv run`

适合你已经创建好模块项目，并且在项目里执行过 `uv sync` 之后。

```bash
uv run crawler4j list
```

对新手来说，最常用的通常是：

1. 用 `uvx --from ...` 创建模块
2. 进入模块目录后，用 `uv run crawler4j ...` 继续补任务和工作流

这也是后文默认采用的命令形式。

## 你还需要一个可运行的宿主应用

只创建模块项目还不够。要验证模块真的能被 `crawler4j` 接受，你还需要：

1. 打开桌面应用
2. 进入模块管理
3. 能添加开发模块（DevLink）
4. 能创建或选择关联策略和作业
5. 能在 ATM 中发起调试或运行

如果你没有宿主应用，只能完成“写模块项目”这一步，无法完成“真实运行与正式交付验收”。

### 为什么宿主应用必须提前准备好

因为很多关键验证都必须在宿主里完成，例如：

- 宿主能否识别 `module.yaml`
- 模块是否能注册成 `DevLink`
- 策略是否能解析到你的模块
- ATM 是否会出现 `🐞 调试`
- zip 安装后模块是否真的来自安装目录

这些都不是模块项目自己能验证出来的。

## 开发前的两条默认约束

### 不要把模块项目虚拟环境当成宿主运行环境

模块项目里 `uv sync` 的作用是让你本地能用 CLI、能跑 import、能写代码。  
它不等于宿主已经有这些依赖。

新手常见误区是：

```text
我在模块目录里 import 成功
=
宿主里也一定能 import 成功
```

这个等式是不成立的。

### 不要先手写脚手架

第一次开发模块时，强烈建议先用 CLI 生成骨架，再去改生成结果：

```bash
crawler4j init-model
crawler4j new
crawler4j add-workflow
crawler4j add-ui
```

这样你得到的是和当前 Core 实现更一致的起点。

## 建议你在继续之前确认这 4 件事

继续读下一页之前，建议确认：

1. 你能执行 `crawler4j --help`
2. 你知道宿主应用在哪里打开
3. 你知道模块最终要在宿主里验证
4. 你接受“先用 CLI 生成骨架，再修改生成结果”这条原则

这 4 件事确认好，后面的成功率会高很多。
