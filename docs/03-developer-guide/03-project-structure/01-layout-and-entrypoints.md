# 3.1 目录结构与入口

## 标准模块项目结构

CLI 生成的标准模块项目，默认结构如下：

```text
hotel_demo/
├── __init__.py
├── .gitignore
├── .python-version
├── pyproject.toml
├── README.md
├── module.yaml
├── config_schema.json
├── tasks/
│   ├── __init__.py
│   └── example_task.py
└── workflows/
    ├── __init__.py
    └── main_workflow.py
```

第一次看这个结构时，你可以先不要想着“每个文件都懂透”。先抓住一个核心思路：

> 这个目录里既有“给人看的文件”，也有“给宿主运行时看的文件”，而真正决定模块能不能运行的，是后者。

## 每个文件真正负责什么

### `module.yaml`

模块清单。它负责声明：

- 模块名
- 版本
- 显示名
- SDK 兼容范围
- 工作流清单
- UI 扩展信息

宿主扫描模块时，先看的是这个文件。

如果这个文件缺失、为空或者关键信息写错，宿主根本还没有机会运行你的 Python 代码。

### 根 `__init__.py`

这是 Core 最终导入的根入口。  
CLI 模板会在这里自动完成两件事：

1. 扫描 `tasks/` 下的 `TaskScript`
2. 扫描 `workflows/` 下的 `TaskFlow`

然后由根 `run(context)` 根据配置决定：

- 调一个工作流
- 直接调一个任务脚本
- 或回退到默认工作流

所以根 `__init__.py` 不是可有可无的装饰文件，而是运行时真正入口。

新手最容易误会的是：“我把任务脚本写在 `tasks/` 里就好了，根文件无所谓。”实际上不是。宿主首先进的是根入口，然后才会再去调任务或工作流。

### `tasks/`

这里放原子任务脚本。每个任务脚本通常只做一个明确动作，例如：

- 登录
- 打开页面
- 采集数据
- 提交结果

对第一次开发模块的人，建议一个任务脚本只做一个清晰动作。不要在一个任务脚本里既登录、又抓取、又提交、又做复杂循环。

### `workflows/`

这里放工作流编排器。工作流不负责细节操作，主要负责：

- 调哪些子任务
- 以什么顺序调
- 循环到什么时候停止
- 在 `ctx.state` 中记录什么阶段信息

简单理解：

- 任务脚本是“做事的人”
- 工作流是“安排步骤的人”

### `config_schema.json`

这是 CLI 生成的声明式配置 UI 文件入口。第一次开发模块时，你可以把它理解成“给模块提供配置表单的最小入口”。

你不需要一开始就把它写得很复杂。先让它能表达几个基础运行参数就够了。

### `pyproject.toml`

这个文件只负责模块项目自己的开发环境。它不等于宿主应用的运行依赖清单。

这句话值得再重复一次，因为它太容易被忽略：

> 模块项目自己的依赖，不会天然自动同步到宿主运行环境。

## 运行时调用顺序

从模块级别看，宿主的执行顺序是：

```text
prepare_env
-> init_env
-> before_run
-> run
-> on_success / on_failure / on_timeout
-> on_cleanup
```

其中：

- `run(context)` 是唯一硬性必需入口
- 其它 hooks 都是可选的

如果 `run(context)` 内部又调到了 `TaskScript`，单个任务脚本的生命周期则是：

```text
on_init
-> execute
-> on_error（仅异常时）
-> on_cleanup
```

理解这两层生命周期非常重要。模块根 hooks 是“模块级运行链路”，`TaskScript` hooks 是“单个任务脚本级运行链路”，不要混为一谈。

### 这对新手意味着什么

如果你在调试时看到某段逻辑没有执行，先判断它属于哪一层：

- 它是在根 `__init__.py` 的 hooks 里吗？
- 还是在某个 `TaskScript` 的生命周期里？

这一点分清楚了，排错会快很多。

## 命名建议

从运行时角度，不是所有名称都必须完全一致；但从维护成本角度，强烈建议保持下面三者一致：

1. 模块目录名
2. Python 包名
3. `module.yaml.name`

例如：

```text
hotel_demo/
module.yaml.name = hotel_demo
```

这样在 DevLink、策略配置、调试、安装和错误排查时最不容易出错。

## 第一次开发模块时，哪些文件最不能随便动

如果你是小白，建议先不要随便删除或重构下面这些文件：

1. 根 `__init__.py`
2. `module.yaml`
3. `tasks/__init__.py`
4. `workflows/__init__.py`

原因很简单：这些文件不是“为了好看”，而是直接参与模块发现、导入和运行。

## 看完这一页后的最小结论

这一页看完后，至少记住下面三句话：

1. 模块目录结构本身就是契约的一部分。
2. 根 `__init__.py` 才是宿主最终进入的运行入口。
3. `pyproject.toml` 属于模块开发环境，不等于宿主运行环境。
