# 1.2 当前真实约束

本节不是“最佳实践建议”，而是当前仓库已经验证过、你必须接受的真实约束。

对于第一次接触项目的人，这一页有个很重要的作用：它可以帮你区分“这是 bug”还是“这是当前设计本来就这样”。

## 1. CLI 命令还叫 `init-model`

当前正式命令是：

```bash
crawler4j init-model <module_name>
```

虽然命令名里仍然是 `model`，但它生成的是当前正式支持的模块项目骨架。

这意味着你在文档、代码、界面和仓库里会同时看到 `model` 和 `module` 两种说法。第一次阅读时不要被它们吓住，当前开发语境下可以把它们理解成同一类东西。

## 2. Core 识别模块看的是目录契约，不是 wheel 元数据

Core 扫描与加载模块时，关键依据是：

- 模块根目录里的 `module.yaml`
- 模块根目录里的 `__init__.py`

因此你不要把“能 build wheel”误当成“宿主能识别模块”。

对新手来说，这条约束的实际含义是：

- 你首先要保证模块目录长得对
- 然后再考虑它是否能被打包

顺序不要反过来。

## 3. 正式安装当前只支持 zip

当前模块管理的正式安装链路认的是 zip 包，不是 `.whl`。  
如果你要做真实交付验收，必须走 zip 安装，而不是只在模块项目本地执行脚本。

所以如果你现在的目标是“交给别人安装”，你不能停在“我本地 `uv run` 成功了”。

## 4. 模块代码最终运行在宿主 Python 环境里

开发时你会在自己的模块项目目录里执行 `uv sync`、`uv run crawler4j ...`。  
但真正运行任务、工作流和 hooks 的，是宿主 `crawler4j` 的 Python 环境。

这直接导致下一条约束。

对新手来说，这条最容易造成误判。因为你很容易以为“我的虚拟环境已经没问题了，宿主应该也没问题”。实际上不是这样。

## 5. 模块项目里的第三方依赖不会被宿主自动安装

如果你在模块项目自己的 `pyproject.toml` 里增加第三方依赖，宿主不会自动帮你同步安装。  
所以新增运行时依赖时，你必须确认：

1. 宿主环境里已经具备该依赖。
2. 或者你把依赖逻辑做成随模块源码一起可用的内容。

这是新开发者最容易踩的坑之一：模块在自己项目里 `import` 正常，放进宿主后却报依赖缺失。

如果你只记住一句话，请记住：

> 模块项目的 `pyproject.toml` 主要服务于模块开发环境，不自动代表宿主运行环境。

## 6. 只有 DevLink 模块才能走当前真实调试链

当前调试能力的前提不是“你本地有源码”，而是：

1. 模块已经注册成 `DevLink`
2. 作业已经绑定策略
3. 策略中的 `execution.module` 解析到该 `DevLink` 模块

只有这样 ATM 才会显示 `🐞 调试`，`DebugService` 才会继续往下创建会话。

这也是为什么很多新手会说“我明明能看到模块，为什么没有调试按钮”。“能看到模块”只说明模块被扫描到了，不说明它已经满足调试入口条件。

## 7. `sdk_version_range` 当前应写成简单 `>=x.y.z`

扫描器现在对版本范围的兼容性判断是简化实现，最可靠的是：

```yaml
sdk_version_range: ">=1.0.3"
```

复杂表达式即使写出来，也不建议当成当前交付口径。

第一次开发模块时，不要主动追求复杂版本表达式。对新手来说，“先写对、先兼容”比“写得很花”更重要。

## 8. 第一次开发模块时，不要把复杂 UI 当主线

当前对新模块作者最稳的 UI 路径只有两类：

1. `config_schema.json` 为主的声明式配置
2. `core:data_table:<view_id>` 通用数据表入口

代码型页面 `ui:SomePage` 并不是完全不能用，但它额外受到以下限制：

- `ui_extension.type` 必须是 `micro_app`
- 模块来源必须是 `DevLink` / 内置来源，或者命中 `mms.ui.allowlist`
- 模块里必须有 `ui.py`，并导出对应的 `QWidget` 类

所以第一次开发模块时，先不要把“自定义 UI 页面”作为首个里程碑。

推荐顺序始终是：

1. 先让任务脚本能跑
2. 再让工作流能调到它
3. 再让 DevLink 调试跑通
4. 最后再考虑增强 UI

## 9. 命名不要求绝对一致，但保持一致最省坑

从运行时角度看，目录名、包名、zip 根目录名、`module.yaml.name` 不一定必须完全一致。  
但从开发、调试、安装和交接成本看，最稳的做法仍然是：

- 目录名一致
- Python 包名一致
- `module.yaml.name` 一致

第一次开发模块时，不要主动制造这种偏差。

## 10. 看这页时应该形成的判断习惯

以后当你遇到一个问题时，先问自己一句：

> 这是我代码写错了，还是我碰到了当前系统的既有限制？

如果答案是后者，就不要用改业务代码的方式去“硬撞”它。应该先回到这页，确认自己是不是踩到了这里列出的约束。

## 11. 模块的数据能力只能来自 `ctx.db`

当前模块开发有一个明确工程边界：

- 模块只能使用 Core 注入到 `TaskContext.db` 上的数据能力
- 不能直接连接宿主数据库
- 不能假设宿主会提供 ORM Session、原生 SQLite 连接或私有仓储对象

当前正式支持的数据能力只包括：

1. 数据集查询：`list_records(dataset)`
2. 数据集写入：`replace_records(dataset, records)`
3. 轻量状态：`get_state` / `set_state` / `exists_state`
4. 幂等锁：`acquire_lock` / `release_lock` / `is_locked`

如果你看到旧资料里还有 `DataService`、`ctx.db.storage`、`ctx.db.accounts` 这类说法，请优先以当前 `TaskContext.db` 的真实接口为准。对新模块来说，不要继续沿用旧抽象。
