# 6.5 CLI（模块脚手架与扩展）

SDK CLI 的目标是让模块开发者可以“最小成本创建完整模块项目，并持续补齐任务、工作流与配置 UI”，同时为后续的校验/打包/诊断预留扩展空间。

> 当前实现参考：`crawler4j_sdk/cli/commands.py`
>
> - 安装 SDK 后命令名为：`crawler4j`
> - 注意：仓库内还存在 Core 应用侧 CLI（`src/cli/*`），二者定位不同。

## 6.5.1 需求说明

- MUST 支持初始化一个完整模块项目（可单独安装 crawler4j-sdk 并运行）。
- MUST 支持生成任务脚本模板（减少样板代码）。
- MUST 支持生成工作流模板并同步更新 `module.yaml`。
- MUST 支持生成或补齐 `config_schema.json` 与 `ui_extension`。
- SHOULD 支持列出当前模块项目中的任务脚本（便于选择/调试）。
- SHOULD 定义统一退出码与输出规范（利于 CI/自动化）。
- MAY 提供 validate/build/pack/doctor 等命令（阶段性落地）。

## 6.5.2 命令表（生成/校验/打包/诊断）

### 6.5.2.1 v1.0 已实现命令（以当前实现为准）

| 命令 | 作用 | 关键参数 | 产物 |
| --- | --- | --- | --- |
| `crawler4j init-model <name>` | 初始化完整模块项目 | `-o/--output`, `-f/--force`, `--defaults`, `--no-git`, `--no-install`, `--workflow-name`, `--display-name`, `--description`, `--workflow-display-name`, `--workflow-description`, `--python-version`, `--no-ui` | `.gitignore`、`.python-version`、`__init__.py`、`module.yaml`、`tasks/example_task.py`、`workflows/main_workflow.py`、可选 `config_schema.json` |
| `crawler4j add [name]` | 交互式创建任务脚本 | `-f/--force` | `tasks/<name>.py` |
| `crawler4j new <name>` | 非交互式创建任务脚本 | `-f/--force` | `tasks/<name>.py` |
| `crawler4j list` | 列出当前模块中的任务脚本 | 无 | stdout 列表 |
| `crawler4j add-workflow <name>` | 创建工作流模板并更新 `module.yaml` | `--display-name`, `--description`, `-f/--force` | `workflows/<name>.py` + `module.yaml` |
| `crawler4j add-ui` | 创建或补齐声明式配置 UI | `--title`, `--description`, `-f/--force` | `config_schema.json` + `module.yaml.ui_extension` |

### 6.5.2.2 预留命令（建议）

> 这部分是“架构预留”，不要求当前立刻实现，但建议预先冻结行为语义，避免未来破坏性变更。

| 命令 | 作用 | 说明 |
| --- | --- | --- |
| `crawler4j validate` | 校验模块项目结构 | 语法检查、`module.yaml` 字段检查、依赖检查、配置 schema（如有） |
| `crawler4j build` | 构建可发布产物 | 例如生成 wheel/sdist |
| `crawler4j pack` | 打包为模块可安装格式 | 对齐 Core 的模块安装器（见 07-Modules / Core） |
| `crawler4j doctor` | 诊断环境 | Playwright 浏览器安装、权限、网络连通性等 |

## 6.5.3 输出与退出码规范

### 6.5.3.1 输出规范

- SHOULD：
  - 普通信息输出到 stdout
  - 错误信息输出到 stdout 或 stderr 均可，但必须有清晰前缀（如 `❌`）
  - 对关键产物路径给出明确提示（便于 IDE/脚本继续处理）
  - `init-model` 在默认模式下应通过交互向导收集关键初始化参数

### 6.5.3.2 退出码

- MUST：
  - `0`：成功
  - `1`：可预期失败（参数错误、目标目录已存在且未 `--force`、文件写入失败等）
- SHOULD：未来可扩展细分（如 `2`=校验失败，`3`=依赖缺失），但同一 major 内不得改变既有含义。

## 6.5.4 `init-model` 初始化语义

- MUST：默认进入初始化向导，让模块开发者完成首批参数配置
- MUST：默认生成 `.gitignore`
- MUST：默认生成 `.python-version`
- MUST：默认执行 `git init`
- MUST：默认执行 `uv sync`
- SHOULD：提供非交互模式，供脚本/CI 通过参数和默认值直接创建项目
