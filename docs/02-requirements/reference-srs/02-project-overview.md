# 第 2 章 项目需求概述（Project Overview）

## 2.1 背景与问题域

Crawler4j 面向“桌面端自动化任务运行平台（Automation Runtime）”场景：

- 系统需要在本地/桌面端稳定运行多个自动化任务（TaskScript）与工作流（TaskFlow），并提供 GUI/CLI 的可观测与可运维能力。
- 任务执行依赖“执行环境（Execution Environment）”，执行环境 **不局限于浏览器**：
  - 浏览器类环境：Playwright、本地浏览器、指纹浏览器服务（如 BitBrowser/VirtualBrowser）等。
  - 非浏览器环境：HTTP Client、桌面应用自动化、外部命令/脚本驱动、移动模拟器等（可逐步引入）。
- 平台的主要困难在于“资源治理与可靠执行”，而非“业务流程本身”。业务流程（站点/字段/验证码/风控对抗等）应由 Modules 侧快速迭代。

本系统聚焦解决以下核心问题：

- **执行问题**：如何以统一的契约描述“原子动作（TaskScript）”与“业务编排（TaskFlow）”，并可靠地执行。
- **资源治理问题**：如何管理 Execution Environment 的生命周期与回收（池化/探活/隔离/兜底清理），避免资源泄漏与僵尸进程。
- **调度问题**：如何在多任务并发下做“需求—供给”撮合（任务需求 ↔ 环境能力/并发槽位供给），并支持背压。
- **工程化问题**：如何为模块开发者提供稳定 SDK、脚手架与校验工具，降低实现偏差。
- **可靠性问题**：如何定义一致的错误语义、重试/回退/熔断策略，并支持崩溃后的状态收敛。

## 2.2 目标与非目标

### 2.2.1 系统目标（Goals）

系统目标 MUST/SHOULD 定义如下：

- **G1（MUST）统一契约**：提供清晰稳定的 SDK 契约（TaskScript/TaskFlow/TaskContext/TaskResult），并作为 Modules 与 Core 的唯一交互边界。
- **G2（MUST）资源可控**：Execution Environment（及其底层资源，如进程/临时目录/连接）必须可治理、可回收、可观测；出现异常时可收敛至 fail-safe 状态。
- **G3（MUST）可调度**：支持可配置的并发、优先级与背压策略，实现任务需求与资源供给的撮合。
- **G4（MUST）可恢复**：对可重试失败提供可配置重试与退避；对不可重试失败给出明确错误语义与处置路径；支持崩溃后的状态收敛与残留资源清理。
- **G5（SHOULD）可审计**：关键操作（启动/停止、环境创建/销毁、模块安装/卸载、策略决策）应产生日志与审计事件，并具备可追溯关联（env_id/workflow_id 等）。
- **G6（SHOULD）开发者体验**：提供 CLI/模板/校验，帮助模块开发者快速生成符合规范的标准模块包与脚本工程。

### 2.2.2 非目标（Non-goals）

以下能力明确不作为系统必须达成的目标：

- **NG1**：不承诺对任何特定站点的“长期可用性”或绕过反爬策略的效果；站点对抗属于各业务模块的实现与运营范畴。
- **NG2**：不提供通用的分布式集群调度与多机资源池（本阶段以桌面/单机运行为主）。
- **NG3**：不内置统一“账号/代理/指纹”等商业服务；系统仅提供接入与治理边界。
- **NG4**：不提供通用数据中台/BI；仅提供任务结果与运行状态的持久化与导出能力。

## 2.3 用户角色与典型场景

### 2.3.1 用户角色（Actors）

- **Operator（运行者）**：安装模块、配置任务、启动/停止、查看日志与导出。
- **Module Developer（模块开发者）**：编写模块包（manifest/workflows/tasks/ui），通过 SDK 开发 TaskScript/TaskFlow。
- **Core Maintainer（核心维护者）**：维护 Core 的资源治理、调度、可恢复与 UI Host。
- **QA（测试）**：验收关键用例与失败注入（断网/超时/崩溃恢复/资源不足）。
- **Ops/SRE（运维）**：关注版本发布、数据迁移、日志/诊断、异常回收。
- **Security（安全）**：定义模块来源与受信策略、安全基线（最小权限、危险能力门控）。

### 2.3.2 典型场景（示例）

- **UC-01 运行任务**：在 GUI 选择模块与工作流，配置参数并启动，观察进度与结果。
- **UC-02 并发运行**：同时运行多个任务，系统按策略分配执行环境并做背压。
- **UC-03 失败恢复**：任务异常退出后可重试；程序崩溃重启后自动清理残留环境并恢复可运行状态。
- **UC-04 模块管理**：安装/升级/卸载标准模块包；加载前校验 SDK 版本与边界规则。
- **UC-05 UI 扩展**：模块提供声明式 UI 页面；少量模块提供受信 micro-app 页面。

## 2.4 约束与假设

### 2.4.1 技术约束（Technology Constraints）

- 主要运行形态为桌面端（GUI）+ 本地持久化（SQLite 为主）+ 本地日志。
- Python 生态（依赖管理以 uv 为准）；GUI 侧可使用 PyQt6/qasync。
- 浏览器自动化可选依赖 Playwright；指纹浏览器服务属于可插拔的 Environment Provider。

### 2.4.2 安全与隔离假设（Security Assumptions）

- Modules 的交付形态为“标准模块包”，来源必须可控；默认视为受信分发。
- Modules 与 Core 的交互 **不允许例外**：Modules 只能依赖 SDK，不得导入 Core 内部（如 `src.*`）。
- 代码 UI micro-app 属于“受信扩展”，必须经过信任门控与能力令牌控制；默认推荐使用声明式 UI。
