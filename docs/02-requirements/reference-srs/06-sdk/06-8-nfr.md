# 6.8 非功能需求（SDK）

本章定义 SDK 作为独立发布物的非功能性约束：稳定性、兼容性、安全性、可观测性、性能与可维护性。

## 6.8.1 稳定性等级（API Lifecycle）

SDK 的对外能力按稳定性分级，便于控制变更风险。

### 6.8.1.1 stable

- MUST：同一 major 版本内向后兼容。
- MUST：行为语义稳定（不仅是签名）。
- SHOULD：有单元测试/契约测试覆盖。

### 6.8.1.2 experimental

- MAY：在 minor 版本中发生破坏性变化（但必须在 Release Notes 明确标注）。
- SHOULD：提供替代方案或迁移路径。

### 6.8.1.3 deprecated

- MUST：给出替代 API。
- MUST：至少跨 1 个 minor 版本保留（推荐更久），并在文档中标记移除计划。

## 6.8.2 版本与兼容策略

### 6.8.2.1 语义化版本（SemVer）

- MUST：遵循 SemVer（MAJOR.MINOR.PATCH）。
  - MAJOR：破坏性变更
  - MINOR：向后兼容的新功能
  - PATCH：向后兼容的修复

### 6.8.2.2 Python 版本

- MUST：`requires-python >= 3.12`（以 `crawler4j_sdk/pyproject.toml` 为准）。
- SHOULD：类型标注完整，保证 IDE 补全与静态检查可用。

### 6.8.2.3 依赖策略

- SHOULD：依赖尽量少、尽量稳定。
- SHOULD：对外可见的对象类型（尤其是 ctx.page/ctx.context）一旦稳定，避免在 minor 版本中替换为不兼容类型。

## 6.8.3 安全与合规

- MUST：不在日志、TaskResult、持久化数据中输出明文密码/Cookie/Token。
- MUST：PII（手机号、证件号等）默认脱敏。
- SHOULD：为外部请求（HTTP/浏览器）设置超时与重试边界，避免资源耗尽。

## 6.8.4 可观测性

- MUST：通过 `ctx.logger` 输出关键阶段日志（开始/结束/关键分支/失败）。
- SHOULD：错误携带可机器判定的 error_code（建议放在 `TaskResult.data.error_code`，见 6.7）。
- SHOULD：对重要输出物（截图、导出文件）提供可追溯引用（path/id）。

## 6.8.5 性能与资源

- SHOULD：脚本执行过程中避免内存无限增长（例如 captured_data 需限量或分页落库）。
- SHOULD：避免在 async 代码里进行长时间阻塞调用。
- SHOULD：对浏览器/HTTP 资源的创建与释放遵循生命周期（能在 on_cleanup 中释放的尽量释放）。

## 6.8.6 文档与示例

- MUST：stable API 必须有文档与示例。
- SHOULD：每个核心抽象（TaskScript/TaskFlow/TaskContext/TaskResult/CLI）都提供“最小可运行示例”。
