# 6.7 错误处理与可靠性（SDK 视角）

本节定义 SDK 层面的错误语义与可靠性建议，确保：

- 脚本作者知道“什么时候 return fail、什么时候 raise”
- 运行时可以做一致的重试/恢复/告警
- 错误信息对人可读、对机器可判定（错误码/可重试标记）

## 6.7.1 错误模型对齐

### 6.7.1.1 错误来源分类

1. **业务失败（可预期）**

- 例：登录失败、验证码错误、页面结构变化导致找不到元素（可归类为业务失败/可恢复）
- 推荐表达：返回 `TaskResult.fail(message, error=...)`

2. **可恢复系统错误（通常可重试）**

- 例：网络超时、短暂 5xx、页面临时加载失败
- 推荐表达：
  - 方案 A（推荐，易于运行时统一处理）：抛出异常并让运行时捕获并决定重试
  - 方案 B：返回 `TaskResult.fail(...)`，并在 `data.retryable=true` 或 `data.error_code=...` 指示

3. **不可恢复错误（通常不应重试）**

- 例：参数校验失败、权限不足、账号被封且需要人工介入
- 推荐表达：返回 `TaskResult.fail(...)`，并给出稳定错误码

4. **缺陷/编程错误（Bug）**

- 例：NoneType 错误、断言失败
- 表达：允许抛出异常；运行时应兜底转换为失败结果并记录堆栈

### 6.7.1.2 错误码（Error Code）规范（建议）

> 目标：让 UI/告警/自动化策略能基于错误码做分流。

- MUST：错误码是稳定字符串，**同一 major 版本内语义不变**。
- SHOULD：采用前缀分域，例如：
  - `SDK-VALIDATION-*`（输入/配置校验）
  - `SDK-NETWORK-*`（网络/HTTP）
  - `SDK-BROWSER-*`（浏览器/页面）
  - `SDK-AUTH-*`（鉴权/登录）
  - `SDK-DATA-*`（数据访问）

推荐承载方式：放入 `TaskResult.data.error_code`（保持 TaskResult 字段稳定，向后兼容）。

### 6.7.1.3 信息分层（给人看 vs 给机器看）

- `TaskResult.message`：给人看的短句（可直接展示在 UI）
- `TaskResult.error`：给人看的错误细节（仍需脱敏）
- `TaskResult.data.error_code`：给机器判定
- `TaskResult.data.artifacts`：截图/日志片段等“证据”引用

## 6.7.2 重试建议与可恢复执行

### 6.7.2.1 重试的基本原则

- MUST：只有在“幂等或可安全重试”的步骤上重试。
- SHOULD：对外部副作用（提交表单、下单、写库）设置幂等键或在执行前做去重校验。
- SHOULD：重试采用指数退避 + 抖动（由运行时实现）。

### 6.7.2.2 在 SDK 层建议脚本作者做到什么

- SHOULD：将流程拆成小步骤并在 `ctx.state["phase"]` 标记阶段，便于失败定位。
- SHOULD：关键游标（分页/最后处理 ID）写入 `ctx.state["cursor"]`，并在适当时机持久化（见 6.6）。
- SHOULD：遇到“需要人工介入”的错误时调用 `ctx.request_stop()` 或返回不可重试的失败码。

### 6.7.2.3 运行时侧对 SDK 的期望（接口层约束）

- MUST：捕获脚本抛出的异常并：
  1. 记录堆栈与环境上下文
  2. 调用 `on_error` 钩子（若存在）
  3. 产出一个失败结果（TaskResult.fail 或等价模型）
- SHOULD：对超时、网络错误、临时页面错误按策略重试。

### 6.7.2.4 截图与证据

- SHOULD：在关键失败点（如 on_error）调用 `await ctx.screenshot(...)`。
- MUST：截图文件名不包含敏感信息（手机号/身份证等）。
