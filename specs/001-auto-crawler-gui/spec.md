# Feature Specification: Automated Crawler with GUI

**Feature Branch**: `001-auto-crawler-gui`
**Created**: 2025-12-24
**Status**: Draft
**Input**: User description provided in conversation.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 自动化任务循环 (Priority: P1)

用户希望系统能自动从携程号码池和劳保号码池获取号码，调度指纹浏览器环境，完成"登录-领题-搜索-提交"的全套流程，并循环执行，实现无人值守的自动化作业。系统应支持配置并发数量（如同时运行 10 个环境），提高整体处理效率。

**Why this priority**: 这是核心业务价值所在，没有自动化流程，工具就失去了意义。

**Independent Test**: 可以使用模拟的号码池和模拟的浏览器环境（或手动控制的一个环境），验证整个流程是否能跑通一次循环。

**Acceptance Scenarios**:

1. **Given** 携程号码池和劳保号码池中均有可用且未在运行的号码，且指纹浏览器环境空闲, **When** 启动自动化任务, **Then** 系统自动打开浏览器环境，分别使用对应账号登录携程和劳保平台，领取任务并完成搜索提交，最后关闭环境。
2. **Given** 上一个任务完成, **When** 自动化任务处于运行状态, **Then** 系统应自动从号码池获取下一个号码，重复上述流程。

---

### User Story 2 - GUI 监控与控制 (Priority: P2)

用户希望通过 PC 桌面客户端界面查看当前运行的环境状态、实时日志，并能手动开始或停止任务。

**Why this priority**: 提供可视化的控制和监控手段，确保用户能掌握程序运行情况，方便运维和干预。

**Independent Test**: 启动 GUI，模拟后端发送的状态更新和日志，验证界面是否正确显示；点击开始/停止按钮，验证是否触发相应的控制逻辑。

**Acceptance Scenarios**:

1. **Given** 程序正在运行, **When** 用户打开 GUI, **Then** 界面应显示当前活跃的浏览器环境列表及其状态（如：登录中、搜索中、提交中）。
2. **Given** 账号正在运行任务, **When** 任务状态更新, **Then** 界面应实时显示该账号的统计数据：已完成题数、废弃题数、审核通过数、审核未通过数。
3. **Given** 有新的日志产生, **When** 用户查看日志区域, **Then** 界面应实时滚动显示最新的操作日志。

---

### User Story 3 - 配置管理 (Priority: P3)

用户需要在界面上导入号码池数据（分为携程账号池和劳保账号池），选择使用的指纹浏览器类型（BitBrowser 或 VirtualBrowser），并设置登录密码等全局配置。

**Why this priority**: 方便用户初始化数据和配置，减少手动修改配置文件的繁琐。

**Independent Test**: 在界面输入配置并导入数据，验证数据是否正确保存到系统存储中。

**Acceptance Scenarios**:

1. **Given** 用户手头有一批携程账号数据, **When** 用户在携程账号页选择导入文件, **Then** 系统应成功解析并将其存入携程号码池。
2. **Given** 用户手头有一批劳保账号数据, **When** 用户在劳保账号页选择导入文件, **Then** 系统应成功解析并将其存入劳保号码池。
3. **Given** 用户需要选择指纹浏览器, **When** 用户在设置页选择 BitBrowser 或 VirtualBrowser, **Then** 系统应检测所选浏览器是否已安装并显示检测结果。
4. **Given** 用户需要修改默认登录密码, **When** 用户在设置页更新密码, **Then** 后续的自动化登录应使用新密码。

---

## Edge Cases

- **浏览器环境启动失败/崩溃**: 系统应能检测到环境异常，尝试重启或标记该环境/账号为失败，并继续处理下一个。
- **号码池耗尽**: 当没有可用号码时，系统应暂停任务并通知用户，而不是无限空转报错。
- **登录失败/验证码**: 虽然描述未详述验证码处理，但若登录失败（如密码错误），系统应记录错误并跳过该账号。
- **劳保平台无题**: 若领题失败或无题可领，系统应等待一段时间或直接结束当前轮次。
- **浏览器未安装**: 若用户选择的浏览器未安装，系统应提示用户并阻止启动任务。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 集成 BitBrowser 或 VirtualBrowser 的 API 以控制浏览器窗口的开启与关闭，用户可在界面选择使用哪种浏览器。
- **FR-002**: 系统 MUST 维护两个独立的号码池：携程账号池和劳保账号池，各自支持 FIFO（先进先出）调度策略。
- **FR-002a**: 系统 MUST 在用户选择浏览器类型后，检测该浏览器是否已安装并显示检测结果。
- **FR-003**: 系统 MUST 使用 Playwright 自动化操作浏览器页面，包括元素点击、输入框填写、内容提取（DOM 解析）。
- **FR-004**: 系统 MUST 实现携程网站和劳保平台的自动登录逻辑。
- **FR-005**: 系统 MUST 实现跨页面的数据流转，将劳保平台的任务关键词在携程搜索，并将结果回填。
- **FR-006**: GUI MUST 使用 PyQt6 构建，并在 Windows/macOS 上运行。
- **FR-007**: GUI MUST 提供可视化的环境列表，展示每个环境的 ID、当前关联账号、运行状态。
- **FR-008**: GUI MUST 提供日志控制台，显示带有时间戳的操作日志。
- **FR-009**: 系统 MUST 支持配置最大并发环境数量（例如同时运行 10 个窗口）。
- **FR-010**: GUI MUST 展示每个劳保账号的详细任务统计：已完成数、废弃数、审核通过数、审核未通过数。
- **FR-011**: 系统 MUST 确保同一时间一个劳保账号只能被一个环境使用。在启动环境时，MUST 仅从状态为"待用"的账号中选择，并在使用期间将其标记为"使用中"。

### Key Entities

- **CtripAccount**: 携程账号。包含手机号、密码、状态 (active/blacklisted/disabled)、接码平台配置 (URL/Key)。
- **LaborAccount**: 劳保账号。包含手机号、密码、状态、任务统计 (completed/discarded/approved/rejected)。
- **Environment**: 绑定一个携程账号 + 一个劳保账号 + 浏览器配置 ID。携程账号被封时删除环境。
- **调度规则**: 从携程账号出发，有环境则启动，无环境则配对空闲劳保账号创建环境。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 自动化流程成功率达到 90% 以上（排除账号本身无效或平台宕机的情况）。
- **SC-002**: 单个账号的处理周期（从启动环境到关闭）平均不超过 3 分钟。
- **SC-003**: GUI 界面响应流畅，状态更新延迟不超过 1 秒。
- **SC-004**: 系统能够连续稳定运行 24 小时无致命崩溃。
